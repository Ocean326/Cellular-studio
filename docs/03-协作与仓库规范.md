# trajectory_annotation_studio 协作与仓库规范

## 1. 仓库定位

本仓是 `Cellular-studio` 的代码主仓，专门管理：

- 标注 studio 的前端与服务端
- 批次接入与适配脚本
- 协作文档、部署文档、合同文档

它不负责保存真实数据、批次结果和 review 产物。

## 2. 当前目录约定

当前真实目录边界固定为：

- 仓库根：`trajectory_annotation_studio/`
- 前端与服务：`trajectory_annotation_studio/web/`
- 脚本：`trajectory_annotation_studio/scripts/`
- 适配器：`trajectory_annotation_studio/adapters/`
- 文档：`trajectory_annotation_studio/docs/`
- 本地数据：`trajectory_annotation_studio/data/`，默认不入 Git

说明：

- `web/` 只放共享逻辑，不接受“某个人数据专用”的硬编码
- `adapters/` 放数据接入差异和转换逻辑
- `data/` 只作为本机临时运行目录，服务器真实数据仍建议放到仓外共享路径

## 3. Git 基本规则

默认分支固定为：

- `main`

`main` 的要求：

- 始终可启动
- 不依赖真实数据也能读懂文档和代码结构
- 不包含私有路径、密码、token、真实批次数据

建议分支前缀：

- `feature/`
- `fix/`
- `docs/`
- `adapter/`
- `spike/`

示例：

- `feature/time-scrubber-polish`
- `fix/review-writeback-error`
- `docs/github-server-governance`
- `adapter/team-zhang/signal100`
- `spike/ref-real-workflow`

## 4. Worktree 规则

推荐使用方式：

```bash
cd trajectory_annotation_studio
git worktree add ../trajectory_annotation_studio-timeline feature/time-scrubber-polish
```

约定：

- 一个 `worktree` 只负责一个明确主题
- 不在同一个 `worktree` 里混做多个大需求
- 合并前回看合同与文档，不各自改出两套数据模型

## 5. 批次命名规则

批次命名建议固定为：

```text
YYYYMMDDTHHMMSSZ_v{version}_{keywords}
```

示例：

- `20260408T070420Z_v2_linefix_local_raw9995`
- `20260410T150000Z_v1_multiday_smoke50`

要求：

- 带时间戳
- 带版本号
- 关键词描述来源、用途或重要特征

## 6. 变更分流规则

任何变更先判断属于哪一类：

### 6.1 核心能力改动

例如：

- 时间轴交互
- review 写回逻辑
- 批次发现机制
- 导出协议

这类改动可以进 `web/` 或 `scripts/`，但必须同步更新文档。

### 6.2 数据接入改动

例如：

- 某同学数据字段不同
- 某数据源目录结构不同
- 需要新建批次转换脚本

这类优先进入：

- `adapters/`
- 独立转换脚本

而不是直接改核心页面逻辑。

### 6.3 本地私有改动

例如：

- 绝对路径
- 本地测试脚本
- 临时调试数据

这类不进主干，留在本地忽略文件里。

## 7. 文档先行规则

满足以下任一条件时，必须先更新 `docs/`：

1. 改了批次目录结构
2. 改了 CSV / manifest / review 输出合同
3. 改了 GitHub 与服务器协作方式
4. 改了“哪些东西可以进 Git，哪些不能进 Git”

## 8. 提交建议

每次提交尽量只做一种类型：

- 纯文档
- 纯接口
- 纯交互
- 纯数据适配
- 纯测试或校验

避免一次提交同时改：

- 标签模型
- 时间轴交互
- 导出逻辑
- 多种数据源适配

否则后续回滚与审阅会很痛。

## 9. 合并前检查

合并前至少确认：

- 没有把真实数据放进 Git
- 没有把个人路径写进共享代码
- 新数据接入是否应该放到 `adapters/`
- 文档索引是否已更新
- 本地最小验证已通过
