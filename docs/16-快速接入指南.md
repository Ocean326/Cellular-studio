# 快速接入指南

## 1. 这份文档给谁看

这份文档面向两类人：

- 组内同学
- 受控 agent / 外部方法接入者

目标很简单：

- 准备一个标准批次
- 上传给 operator
- 在 studio 中被挂出来
- 后续能拿回自己的标注数据

## 2. 你需要准备什么

最小标准结果目录：

```text
my_batch/
  result/
    manifest.json
    states_index.json
    <uid_1>/
      raw.csv
      line.csv 或 fmm.csv
    <uid_2>/
      raw.csv
      line.csv 或 fmm.csv
```

更推荐的完整结构：

```text
my_batch/
  result/
    manifest.json
    states_index.json
    <uid>/
      raw.csv
      snap.csv
      od.csv
      fmm.csv
      line.csv
```

关键要求：

- `manifest.json` 里必须有 `uids`
- `states_index.json` 必须存在
- `result/<uid>/` 目录必须真实存在
- 每个 UID 至少要有 `raw.csv`
- 如果要进入人工审阅，最好至少有 `line.csv` 或 `fmm.csv`

## 3. 上传方式

如果 operator 开了 Web 上传入口，直接把结果目录打成 zip 上传。

如果 operator 给的是 HTTP 上传口，也可以直接用：

```bash
cd my_batch
zip -qr ../my_batch.zip result

curl -X POST "http://127.0.0.1:8016/api/admin/incoming/upload?name=my_batch.zip" \
  -H 'Content-Type: application/zip' \
  -H 'X-Upload-Filename: my_batch.zip' \
  --data-binary @../my_batch.zip
```

推荐 zip 内部结构为：

```text
result/
  manifest.json
  states_index.json
  <uid>/
```

不要把一堆无关缓存文件塞进去，例如：

- `.DS_Store`
- `__pycache__/`
- 大体积训练缓存
- 临时日志

## 4. 被接入后会发生什么

operator 会做：

1. 把你的 zip 放进 `incoming`
2. 解包并校验结构
3. 如通过，则发布为一个 studio 批次

当前是 `operator-controlled publish`，不是上传后自动上架。

## 5. 我如何知道自己的批次有没有挂出来

operator 发布后，会给你一个批次名，例如：

```text
incoming_demo_b01
```

之后可以在 studio 的批次列表中找到它。

## 6. 如何拿回我的标注结果

当前支持 `reviewer bundle export`。

导出内容包含：

- 你自己的审核记录
- 你自己的 timeline annotations
- 对应样本的原始工件

如果需要 operator 帮你导出，至少提供：

- `batch`
- `reviewer_id`

例如：

```json
{
  "batch": "incoming_demo_b01",
  "reviewer_id": "alice",
  "clean": true,
  "bundle_name": "alice_bundle",
  "create_zip": true
}
```

## 7. 常见失败原因

### 7.1 没有 `manifest.json`

系统无法知道有哪些 UID。

### 7.2 `manifest.json.uids` 和真实目录不一致

会导致校验失败，或者批次打开后部分 UID 不可用。

### 7.3 没有 `line.csv` 或 `fmm.csv`

如果 operator 在 intake 时启用了 `--require-review-reference`，这种会被拦下。

### 7.4 想直接把数据扔进 `published`

不允许。当前流程固定为：

```text
incoming -> validate -> publish
```

## 8. 推荐你的自检命令

如果你本地也有这份仓库，先自检：

```bash
cd my_batch
zip -qr ../my_batch.zip result

mkdir -p /tmp/trajectory_annotation_studio/incoming_upload_demo
cp ../my_batch.zip /tmp/trajectory_annotation_studio/incoming_upload_demo/payload.zip

cat > /tmp/trajectory_annotation_studio/incoming_upload_demo/upload_status.json <<'JSON'
{"upload_id":"demo_upload","status":"uploaded","original_name":"my_batch.zip"}
JSON

python3 scripts/process_incoming_upload.py \
  --upload-root /tmp/trajectory_annotation_studio/incoming_upload_demo \
  --workspace-root /tmp/trajectory_annotation_studio/incoming_workspace \
  --require-review-reference \
  --clean-workspace
```

如果你只是想让 operator 验证一个已发布批次结构：

```bash
python3 scripts/validate_published_batch.py \
  --batch-root /path/to/published_batch \
  --require-review-reference
```

## 9. 当前边界

当前版本已经支持：

- 上传 zip
- operator 校验
- operator 发布
- reviewer bundle 导出

当前还不支持：

- 上传后自动上架
- 复杂审批流
- 多文件断点续传
- 服务端统一标签治理
