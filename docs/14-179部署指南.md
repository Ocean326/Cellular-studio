# 179 部署指南

## 1. 适用范围

这份文档面向 `179` 上的首次部署或新 release 部署。

目标是把 `Cellular-studio` 以单实例方式部署到：

```text
/srv/trajectory_annotation_studio
```

当前默认姿态固定为：

- 单实例 `review_server.py`
- `nginx` 反代
- `systemd` 托管
- 共享批次从 `shared/published` 发现
- 上传暂存进入 `shared/incoming`

## 2. 前置条件

服务器需具备：

- `python3`
- `nginx`
- `systemd`
- 一个可写的部署根目录 `/srv/trajectory_annotation_studio`

建议的系统身份：

- 服务用户：`ta_svc`
- 管理组：`ta_admin`
- 上传组：`ta_uploaders`

## 2.1 如果当前账号没有 `sudo`

这不是异常情况，而是首轮经常会遇到的真实约束。

如果像本轮 `179` 一样：

- 可以 SSH 登录
- 可以写自己的 home 目录
- 但没有免密 `sudo`

那就先走 `user-space pilot`，不要卡在 `/srv + systemd + nginx`。

推荐把部署根切到：

```text
/home/<user>/trajectory_annotation_studio_pilot
```

对应变化只有三类：

- 根目录从 `/srv/trajectory_annotation_studio` 改成 `/home/<user>/trajectory_annotation_studio_pilot`
- 服务先用 `nohup python3 .../web/review_server.py` 起在高位端口，比如 `18016`
- 首轮访问通过 `ssh -L` 隧道完成，而不是直接走域名

这条路径适合：

- 首轮真实 smoke
- 小样本验证
- operator 先把发布流跑通

不适合直接宣称“正式上线”。

## 3. 初始化服务器目录

在服务器上执行：

```bash
sudo mkdir -p /srv/trajectory_annotation_studio
sudo chown "$USER" /srv/trajectory_annotation_studio

cd /srv/trajectory_annotation_studio
python3 /path/to/trajectory_annotation_studio/scripts/init_179_server_layout.py \
  --root /srv/trajectory_annotation_studio \
  --release-name bootstrap \
  --create-current-link
```

初始化后应得到：

```text
/srv/trajectory_annotation_studio/
  app/
    releases/
    current
  datasets/
    cellular_quality/
  shared/
    published/
    incoming/
    exports/
  runtime/
    logs/
    tmp/
    uploads/
  workspaces/
```

如果当前没有 `sudo`，把上面的 `/srv/trajectory_annotation_studio` 全部替换成：

```text
/home/<user>/trajectory_annotation_studio_pilot
```

## 4. 准备代码 release

推荐每次部署一个新的 release 目录，不直接在 `app/current` 里改代码。

### 4.1 从 GitHub 拉代码

第一次部署：

```bash
cd /srv/trajectory_annotation_studio/app/releases
git clone git@github.com:Ocean326/Cellular-studio.git 20260415_r1
```

后续新 release：

```bash
cd /srv/trajectory_annotation_studio/app/releases
git clone git@github.com:Ocean326/Cellular-studio.git 20260415_r2
```

### 4.2 切换 `app/current`

```bash
cd /srv/trajectory_annotation_studio/app
rm -f current
ln -s releases/20260415_r1 current
```

## 5. 配置 systemd

仓库里已有模板：

- [trajectory_annotation_studio.service.example](/Users/ocean/Documents/Playground/Cellular-projects/trajectory_annotation_studio/deploy/systemd/trajectory_annotation_studio.service.example)

推荐步骤：

```bash
sudo cp /srv/trajectory_annotation_studio/app/current/deploy/systemd/trajectory_annotation_studio.service.example \
  /etc/systemd/system/trajectory_annotation_studio.service
sudo systemctl daemon-reload
sudo systemctl enable trajectory_annotation_studio.service
sudo systemctl restart trajectory_annotation_studio.service
sudo systemctl status trajectory_annotation_studio.service --no-pager
```

模板默认监听：

- `127.0.0.1:8016`

默认挂载：

- `--project-root /srv/trajectory_annotation_studio/app/current`
- `--batches-root /srv/trajectory_annotation_studio/shared/published`
- `--incoming-root /srv/trajectory_annotation_studio/shared/incoming`

如果当前没有 `sudo`，这一节可以先跳过，改为直接起一个用户态后台进程：

```bash
nohup python3 /home/<user>/trajectory_annotation_studio_pilot/app/current/web/review_server.py \
  --host 127.0.0.1 \
  --port 18016 \
  --project-root /home/<user>/trajectory_annotation_studio_pilot/app/current \
  --batches-root /home/<user>/trajectory_annotation_studio_pilot/shared/published \
  --incoming-root /home/<user>/trajectory_annotation_studio_pilot/shared/incoming \
  >/home/<user>/trajectory_annotation_studio_pilot/runtime/logs/review_server_18016.log 2>&1 < /dev/null &
```

然后本机通过隧道访问：

```bash
ssh -L 18016:127.0.0.1:18016 <179-host-or-alias> -N
```

## 6. 配置 nginx

仓库里已有模板：

- [trajectory_annotation_studio.nginx.conf.example](/Users/ocean/Documents/Playground/Cellular-projects/trajectory_annotation_studio/deploy/nginx/trajectory_annotation_studio.nginx.conf.example)

复制后至少修改：

- `server_name`
- TLS 证书路径
- 访问控制方式

示例命令：

```bash
sudo cp /srv/trajectory_annotation_studio/app/current/deploy/nginx/trajectory_annotation_studio.nginx.conf.example \
  /etc/nginx/sites-available/trajectory_annotation_studio.conf

sudo ln -sf /etc/nginx/sites-available/trajectory_annotation_studio.conf \
  /etc/nginx/sites-enabled/trajectory_annotation_studio.conf

sudo nginx -t
sudo systemctl reload nginx
```

## 7. 首次 smoke 检查

先看本机健康状态：

```bash
curl "http://127.0.0.1:8016/api/health?batch=<batch-name>"
curl http://127.0.0.1:8016/api/batches
curl http://127.0.0.1:8016/api/admin/incoming
```

注意：

- 当 `shared/published` 下已经有多个 batch 时，`/api/health` 建议总是显式带 `?batch=<batch-name>`，避免检查到默认批次而不是你刚发布的那个。

然后看网页是否可打开：

```text
https://<your-domain>/web/index.html
```

如果当前还是 `user-space pilot`，则把“浏览器打开域名”替换成：

```text
http://127.0.0.1:18016/web/index.html
```

前提是你已经在本机开了：

```bash
ssh -L 18016:127.0.0.1:18016 <179-host-or-alias> -N
```

## 8. 首轮发布前必须确认

在真正挂 `10天 * 10000 UID` 前，先确认：

- `review_server.py` 当前按单实例运行
- `incoming` 不在 `--batches-root` 下面
- `shared/published` 只有 operator 发布，不给普通上传者写权限
- reviewer bundle 导出目录不写入 Git

## 9. 当前已验证与未验证边界

已经在本地验证过：

- 目录初始化
- `systemd/nginx` 模板存在且参数对齐
- `published batch` 发现
- `incoming -> validate -> publish` CLI

已经在真实 179 上验证过：

- 当前账号可走 `user-space pilot`
- 可以在 `/home/<user>/trajectory_annotation_studio_pilot` 初始化目录
- 可以用用户态 `review_server.py` 挂起服务并通过 `ssh -L` 访问
- 可以挂载真实 smoke batch 并通过 `/api/batches` 与 `/api/health` 验证

还未在真实 `179` 上验证：

- 真 TLS 与鉴权
- 真大批次 I/O 性能
- 真运维账号与权限组

因此这份指南现在分成两层使用：

- `user-space pilot`：已经可直接照着做
- `/srv + systemd + nginx` 正式姿态：仍需要有 sudo 的 operator 再落一次
