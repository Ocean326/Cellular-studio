# 20260420 小版本更新与 179 重部署说明

## 1. 这次版本的定位

这不是单点修 bug，而是一轮面向真实使用链路的小版本升级。

目标有 4 个：

1. 把 Studio 主页面的前端壳体从“大量逻辑堆在 `index.html`”推进到更稳定的模块结构。
2. 把“自主上传 -> 处理 -> 打开 -> 删除”的批次产品化链路补成闭环。
3. 把时间轴、标注条、设置与 Studio 管理的交互体验提升到可持续迭代的状态。
4. 在不改动现有 `179` 数据面的前提下，整理出一版可推送、可重新部署的代码 release。

## 2. 高信号改动汇总

### 2.1 前端架构升级

- `web/index.html` 不再继续独占大部分前端逻辑，运行态代码被拆入：
  - `web/app/runtime/*`
  - `web/app/features/*`
  - `web/styles/index.css`
- 新增稳定 bootstrap 配置层：
  - `web/app/studio_bootstrap.js`
- 当前主要模块边界已经落下：
  - `review_flow`
  - `track_inspection`
  - `studio_admin`
  - `workspace`

这次拆分的意义不是“为了重构而重构”，而是为了让后续 UI、交互、上传与批次能力继续演进时，不再反复把 `index.html` 变成单文件瓶颈。

### 2.2 Studio 主页面体验升级

- 时间轴支持更完整的窗口操作：
  - 全时间窗口联动地图点展示范围
  - `<>` 点位切换
  - 日期窗口“跨度”语义调整为：
    - `0 = 不限`
    - `1 = 1 天`
- 日期窗口支持整段快标、状态回显与快捷操作。
- 做标注后，整轨决策会自动切到“保留”，但不自动保存。
- 标注设置中新增可配置快捷键：
  - 保留
  - 排除
  - 跳过
  - 保存
- 顶部标注条、右下角工具区、设置与提示入口做了多轮紧凑化和响应式对齐优化。
- 主题模式与磨砂风格进一步统一，夜景模式可持久化。

### 2.3 Studio 管理与自主上传闭环

- 主页面新增 `Studio 管理` 浮窗工作区，而不是跳独立上传页面。
- 上传能力已补成完整链路：
  - 创建 upload
  - 上传原始文件
  - 处理成 batch
  - 打开生成批次
  - 删除 upload / soft delete batch
- 支持：
  - `trajectory4`
  - `signal6`
- 上传字段兼容性增强：
  - 字段名大小写不敏感
  - 常见 snake_case / camelCase 变体兼容
  - `signal6` 时间字段支持扩展 alias
  - 支持自定义字段映射
- 增加：
  - 上传进度前端显示
  - 说明 `i` 浮层
  - 上传可见性与 owner scope 逻辑
  - 发布后打开 / 删除后的刷新

### 2.4 批次合同、适配层与后端补强

- 新增统一 batch 合同工具：
  - `batch_contract.py`
- `manifest` 驱动的图层文件与 review reference 解析能力进一步稳定。
- 新增用户上传适配核心：
  - `scripts/user_upload_adapter_lib.py`
  - `web/upload_lib.py`
- review server、导出脚本、批次发布与 intake 校验逻辑同步补强。
- 新增 adapter 模板，后续接新数据优先复制模板而不是改核心逻辑：
  - `adapters/template/`

### 2.5 仓库健康与交付护栏

- 新增最小仓库健康工作流：
  - `.github/workflows/repo-health.yml`
- 包入口和脚本入口的兼容性进一步收敛：
  - `python3 web/review_server.py`
  - `python3 -m trajectory_annotation_studio.web.review_server`
- 增加前端与后端契约测试，覆盖：
  - runtime bootstrap
  - Studio 管理 UI 合同
  - 主题模式
  - 日期窗口快标
  - 上传能力

## 3. 本次对 179 数据面的兼容判断

这次版本可以按“代码 release 升级”处理，不需要迁移或改写现有 `179` 数据目录。

兼容判断依据：

1. `179` 当前运行实例的数据与代码目录已经分离：
   - 代码：`/home/jianghy/trajectory_annotation_studio_pilot/app/current`
   - 共享批次：`/home/jianghy/trajectory_annotation_studio_pilot/shared/published`
   - 上传暂存：`/home/jianghy/trajectory_annotation_studio_pilot/shared/incoming`
   - 数据集：`/home/jianghy/trajectory_annotation_studio_pilot/datasets/cellular_quality`
2. 当前线上批次是 mounted 模式，`batch_meta.json` 通过 `source_result_root` 指向真实结果根。
3. 新代码继续兼容现有：
   - `manifest.json`
   - `states_index.json`
   - `line.csv / fmm.csv` review reference 语义
   - reviewer namespace 目录
4. 这次发布不需要修改：
   - `shared/published/*/review`
   - `shared/published/*/accepted_assets`
   - `shared/incoming/*`
   - `datasets/cellular_quality/*`

换句话说，这次重部署的正确姿态是：

- 新建一个 release 目录
- 切 `app/current`
- 重启 `18016` 与 `8016` 两个 Python 进程

而不是：

- 改 `shared/`
- 改 `datasets/`
- 重写历史 `review`

## 6. 重部署原则

这次重部署固定遵守 3 条：

1. 只换 `app/releases/<release>` 与 `app/current`，不动 `shared/` 和 `datasets/`。
2. 先用新 release 对现有 published batch 做只读验证，再切流。
3. 切流后立即做最小 smoke：
   - `/api/batches`
   - `/api/health?batch=<current_batch>`
   - 页面入口可打开
