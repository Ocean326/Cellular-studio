# trajectory_annotation_studio 后台逻辑与存储说明

这份文档专门回答当前实现里的 3 个核心问题：

1. 自定义数据怎么接入
2. 标注记录怎么存、存到哪里
3. 左侧筛选标签逻辑是什么

## 1. 自定义数据怎么接入

## 1.1 当前推荐入口：`--batches-root`

当前最推荐的接入方式不是改代码，也不是把数据复制到 `web/` 下面，而是启动服务时指定：

```bash
python3 web/review_server.py \
  --host 0.0.0.0 \
  --port 8016 \
  --project-root /path/to/trajectory_annotation_studio \
  --batches-root /path/to/shared/batches
```

这样服务会扫描：

```text
<batches-root>/*/
```

每个一级子目录都视作一个批次候选。

## 1.2 服务端是怎么发现批次的

批次发现逻辑在：

- [review_server.py](/Users/ocean/Documents/Playground/Cellular-projects/trajectory_annotation_studio/web/review_server.py)

核心规则是：

1. 遍历 `batches-root` 下的一级子目录
2. 先看 `<batch_root>/result/` 是否存在
3. 如果 `result/` 不存在，再读 `<batch_root>/batch_meta.json`
4. 如果 `batch_meta.json` 里有 `source_result_root`，就把它当成真实结果目录
5. 只要最终能解析出有效结果目录，该批次就会出现在前端批次列表里

也就是说，自定义数据最重要的是：

- 让服务能找到一个真实的 `result_root`
- 让 `review_root` 和 `accepted_assets` 有固定位置

## 1.3 前端是怎么读 UID 列表的

前端逻辑在：

- [index.html](/Users/ocean/Documents/Playground/Cellular-projects/trajectory_annotation_studio/web/index.html)

读取顺序是：

1. 优先请求 `manifest.json`
2. 如果 `manifest.json` 里有 `uids`，直接用它
3. 如果没有 `manifest.json`，退回目录扫描 `result/<uid>/`

因此：

- 小批次没有 `manifest.json` 也能凑合打开
- 大批次强烈建议提供 `manifest.json`

否则筛选和首屏加载都会变慢。

## 2. 标注记录怎么存、存到哪里

## 2.1 整轨审核

整轨审核通过：

- `GET /api/reviews`
- `POST /api/reviews`

进行读写。

后端逻辑在：

- [review_lib.py](/Users/ocean/Documents/Playground/Cellular-projects/trajectory_annotation_studio/web/review_lib.py)

当前主落盘位置：

```text
<batch_root>/review/
  system/
    reviewer_registry.json
  reviewers/
    <reviewer_id>/
      profile.json
      reviews/
        ledger.jsonl
        latest_reviews.json
  aggregate/
    stats.json
    by_uid/
      <uid>.json
```

存储语义：

- `reviewers/<reviewer_id>/reviews/ledger.jsonl`
  某个 reviewer 自己的 append-only 审核流水
- `reviewers/<reviewer_id>/reviews/latest_reviews.json`
  某个 reviewer 自己的“按 UID 最新结果”
- `aggregate/by_uid/<uid>.json`
  当前 UID 的多人 latest 汇总
- `aggregate/stats.json`
  reviewer 数量、冲突 UID、每个 UID 的 reviewer 计数等缓存摘要

因此“编辑审核结果”的真实语义不是原地覆盖，而是：

- 对“同一个 reviewer + 同一个 UID”再追加一条新 review
- 左侧 triage 只认当前 reviewer 的 latest
- 右侧多人汇总面板会同时展示不同 reviewer 的 latest

## 2.2 时间轴标记

时间轴标记通过：

- `GET /api/timeline-annotations`
- `POST /api/timeline-annotations`

进行读写。

当前主落盘位置：

```text
<batch_root>/review/
  reviewers/
    <reviewer_id>/
      timeline_annotations/
        <uid>.json
        ledger.jsonl
```

存储语义：

- `<uid>.json`
  某个 reviewer 对当前 UID 的最新 `pins + segments` 快照
- `ledger.jsonl`
  该 reviewer 自己的时间轴标记流水

这意味着：

- 时间轴标记不再是“全局单份”
- 同一轨迹上，不同 reviewer 的时间轴标记可以并存
- 当前 reviewer 默认只编辑自己的时间轴标记

## 2.3 `accepted_assets`

`accept` 之后并不会自动复制参考资产，只有显式执行导出时才会生成：

```text
<batch_root>/accepted_assets/reviewers/<reviewer_id>/
```

导出逻辑会：

1. 读取某个 reviewer 的 `latest_reviews.json`
2. 筛出该 reviewer 最新状态为 `accept` 的 UID
3. 复制 `raw.csv`
4. 优先复制 `line.csv`，否则回退复制 `fmm.csv`
5. 同时写出 `review.json`

多人汇总导出会落到：

```text
<batch_root>/review_exports/aggregate/
```

## 2.4 浏览器本地还保存了什么

除了服务端落盘，前端还会在 `localStorage` 里缓存一些 UI 状态：

- reviewer session
- 侧边栏宽度
- 时间轴透明度设置
- 自定义标注类别
- timeline pins / segments 的浏览器缓存副本

但要区分清楚：

- 当前批次的**数据真源**仍然是服务器写回文件
- `localStorage` 只是 UI 体验层缓存与当前 reviewer session 缓存

尤其是时间轴标记，当前实现会：

1. 先写本地内存
2. 同步写 `localStorage`
3. 再按当前 reviewer 异步 POST 到服务端

所以真正用于共享与持久化的，仍然是：

- `<batch_root>/review/reviewers/<reviewer_id>/timeline_annotations/`

## 3. 左侧筛选标签逻辑是什么

## 3.1 标签选项从哪里来

左侧筛选区显示哪些标签，优先级如下：

1. `manifest.json.filter_state_options`
2. 如果没有，就用当前 UI 模式的默认集合

对于 `chain2`，默认通常包括：

- `road`
- `subway`
- `railway`
- `unmatch`
- `stay`

对于 `sim_signal`，默认集合会切换成 GPS / 模拟信令相关状态。

## 3.2 每个 UID 的状态从哪里来

前端判断一个 UID 具有什么状态，优先级如下：

1. `manifest.json.states[uid]`
2. `states_index.json[uid]`
3. 如果都没有，就临时读 CSV 现算

这也是为什么推荐批次提前生成：

- `manifest.json`
- `states_index.json`

## 3.3 `chain2` 模式下怎么现算状态

如果没有预计算状态，前端会按下面规则计算：

- 从 `fmm.csv` 和 `line.csv` 的 `match_type` 推导：
  - `road`
  - `subway`
  - `railway`
  - `unmatch`
- 从 `od.csv.is_stationary` 推导：
  - `stay`
  - `road`

也就是说，`chain2` 下左侧筛选本质上是在读：

- `fmm.match_type`
- `line.match_type`
- `od.is_stationary`

## 3.4 `sim_signal` 模式下怎么现算状态

如果是 `sim_signal` 模式，前端会按下面规则推导：

- 从 `gps.csv.status` 直接读状态
- 从 `signal.csv` 额外推导：
  - `ping_pong`
  - `long_jump`

其中 `long_jump` 会参考：

- `manifest.json.long_jump_threshold_m`

如果没提供，就使用默认阈值。

## 3.5 搜索框匹配什么

当前全局搜索优先站在“当前 reviewer 视角”，会匹配：

- `uid`
- 当前 reviewer 的最新 `notes`
- 预计算状态文本

所以 UID、当前 reviewer 的备注关键词、状态词都可能搜出来。

## 3.6 三列审阅箱怎么分桶

筛选完成后，左侧并不是简单地列 UID，而是按“当前 reviewer 的最新 review”分成 3 列：

- `待审`
  没有最新 review
- `已通过`
  最新 `decision = accept`
- `未通过/跳过`
  最新 `decision in {reject, skip}`

第三列还支持子筛选：

- 全部
- 仅 `reject`
- 仅 `skip`

## 4. 当前需要知道的边界

## 4.1 这是批次级持久化，不是数据库系统

当前实现没有数据库，靠的是：

- 批次目录
- JSON / JSONL
- CSV

优点是简单透明，缺点是：

- 并发冲突控制弱
- 权限体系缺失
- 需要外部做好备份

## 4.2 目前没有服务端统一标签字典

时间轴“新建哪种段落标签”的菜单来源，当前主要还是浏览器本地设置，而不是服务端 `label_catalog.json`。

所以如果后续要做真正组内统一标签体系，需要单独推进：

- 服务端读取标签目录
- 批次启用标签
- 标签版本冻结

## 4.3 旧 `segments/` 目录不是当前主逻辑

如果你在老批次里看到：

```text
<batch_root>/segments/
```

不要把它当成当前页面主逻辑。

当前生效的是：

```text
<batch_root>/review/timeline_annotations/
```
