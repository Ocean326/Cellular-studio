# 标注工具系统升级开发计划

## Summary

本计划把升级拆成“先独立、再兼容、再增强”的 5 个阶段，避免一上来把数据模型、前端交互、导出逻辑同时推翻重做。

核心原则固定为：

- 上游管线仍以 `cellular_quality` 为主
- 新工具优先兼容旧审核资产
- 先做结构性骨架，再做复杂交互

## Phase 0：仓库与设计冻结

目标：

- 建立独立仓库
- 冻结文档与实施边界
- 明确协作方式

产物：

- 仓库骨架
- `docs/` 文档
- 本地 Git 初始化
- `worktree` 协作规范

验收：

- 新仓库可独立管理
- 文档足以指导后续实现

## Phase 1：兼容接入与最小可运行壳

目标：

- 把现有 review UI 的可用能力迁移到新子项目
- 先不改变旧数据语义

实现重点：

- 新工具能读取旧批次
- 保留批次切换、轨迹选择、整轨审核、地图与图层展示
- 为后续后端接口与数据适配层留清晰边界

关键改动：

- 建立前端入口与后端服务入口
- 引入 adapter 层读取 `cellular_quality` 当前结果目录
- 兼容旧 `ledger.jsonl`、`latest_reviews.json` 与 `accepted_assets`

验收：

- 至少一个旧批次能在新工具中打开
- 不改动旧数据也能完成整轨审核读写

## Phase 2：多天长轨迹模型

目标：

- 从 UID 轨迹浏览升级到“person-long-track”浏览

实现重点：

- 构建 `track_manifest`
- 接入 `signal_100+cid2gps.csv`
- 支持跨多天长轨迹
- 支持时间窗口读取

关键改动：

- 数据侧新增 `track_id`、`person_id`、时间跨度、可用图层等元信息
- 页面新增时间段选择器
- 服务端支持按时间窗口返回数据

验收：

- 同一条长轨迹可查看不同时间窗口
- 跨午夜和跨天窗口能正确显示

## Phase 3：审核 V2 与标签体系

目标：

- 保留一级审核
- 加入项目级标签字典与二级标签

实现重点：

- 建立标签字典配置
- 整轨审核写回扩展支持 `trajectory_tags`
- 批次可配置启用标签

关键改动：

- 后端新增标签查询与标签校验
- 前端新增标签选择区与已打标签摘要
- 导出逻辑兼容旧审核结果

验收：

- `accept` 轨迹可写入二级标签
- `reject / skip` 轨迹不能写入二级标签

## Phase 4：时间轴与分段标注

目标：

- 建立真正可用的分段标注工作流

实现重点：

- 地图下方新增时间轴
- 时间轴锚点可切换图层
- 支持起点/终点创建、编辑与删除
- 支持分段标签写回与回显

关键改动：

- 新增 `segment_annotation` ledger
- 前端支持区间创建、光标联动、已有标注高亮
- 服务端新增 segment 读写接口

验收：

- 能对 `accept` 轨迹创建至少一条分段标签
- 重新打开轨迹时能正确回显与编辑

## Phase 5：导出、统计与研究接入

目标：

- 让标注结果真正成为后续研究资产

实现重点：

- 导出整轨审核 + 轨迹标签 + 分段标签
- 提供批次级统计
- 明确哪些资产回流 `research_arena`

关键改动：

- 导出 manifest
- 审核/标签/分段覆盖率统计
- organizer 可随时导出“当前已完成部分”

验收：

- 任意时刻可导出当前完成资产
- 资产带有批次、轨迹、审核人与时间信息

## 公共接口约定

首发建议固定以下接口：

- `GET /api/project-config`
- `GET /api/batches`
- `GET /api/tracks`
- `GET /api/track/:track_id`
- `GET /api/reviews`
- `POST /api/reviews`
- `GET /api/segment-annotations`
- `POST /api/segment-annotations`
- `GET /api/labels`

## Test Plan

必须覆盖以下测试：

1. 旧批次兼容读取
2. 多天窗口切换
3. 一级审核与二级标签权限边界
4. 分段标签创建、编辑、删除
5. 旧审核数据回读不报错
6. 导出结果可追溯
7. `worktree` 并行开发不冲突

## Assumptions

- 首版不做云端部署
- 首版不做多人实时协同
- 首版只允许 organizer 管理标签字典
- 首版分段标签只支持单连续时间区间
