# trajectory_annotation_studio 协作与仓库规范

## 1. 仓库定位

本项目是 `Cellular-projects` 仓内的独立 Git 仓库，目的是：

- 降低对 `cellular_quality` 主项目的直接干扰
- 允许前端、后端、数据适配并行演进
- 支持 `git worktree` 开多个工作副本

## 2. 本地目录约定

固定目录：

- 仓库根：`trajectory_annotation_studio/`
- 文档：`trajectory_annotation_studio/docs/`
- 代码：`trajectory_annotation_studio/src/`
- 配置：`trajectory_annotation_studio/configs/`

## 3. Git 规则

默认分支固定为：

- `main`

建议分支前缀：

- `feature/`
- `fix/`
- `spike/`
- `docs/`

若是 agent 或临时实验，可额外带标识，例如：

- `feature/timeline-segmentation`
- `fix/review-ledger-v2`
- `spike/multi-day-loader`

## 4. Worktree 规则

推荐使用方式：

```bash
cd trajectory_annotation_studio
git worktree add ../trajectory_annotation_studio-timeline feature/timeline-segmentation
```

约定：

- 一个 `worktree` 只负责一个明确主题
- 不在同一个 `worktree` 里混做多个大需求
- 合并前先回看文档和接口约定，不要各自改出两套模型

## 5. 批次命名规则

批次命名固定为：

```text
YYYYMMDDTHHMMSSZ_v{version}_{keywords}
```

示例：

- `20260408T070420Z_v2_linefix_local_raw9995`
- `20260410T150000Z_v1_multiday_smoke50`

要求：

- 必须带 UTC 时间戳
- 必须带版本号
- 关键词尽量描述数据来源、用途或重要特征

## 6. 变更协作规则

任何变更应先对齐以下三件事：

1. 是不是在本仓范围内解决，而不是回写到 `cellular_quality`
2. 有没有改动数据合同
3. 有没有影响旧批次兼容

如果会影响数据合同，必须先更新 `docs/` 再改实现。

## 7. 首阶段协作建议

建议按职责拆成三路并行：

- 前端交互
  负责时间段选择器、时间轴、标签 UI
- 后端与审核资产
  负责 review / segment / labels 接口与 ledger
- 数据适配
  负责多天长轨迹、manifest、窗口化读取

## 8. 提交建议

每次提交尽量只做一种类型：

- 纯文档
- 纯接口
- 纯交互
- 纯数据适配

避免一次提交同时改：

- 标签模型
- 时间轴交互
- 导出逻辑

否则后续回滚和并行协作会很痛。
