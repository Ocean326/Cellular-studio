# 轨迹编辑模式与严格分段导出技术规格

更新时间：`2026-04-27`

## 1. 目标

本规格覆盖三块升级：

1. 完整轨迹编辑模式
2. 按 `gps + timeline 标注段` 的严格导出
3. Studio 导出工作区与相关浮层布局治理

## 2. 术语

- `timeline segment`：时间轴上的标签段
- `exclusive mode`：互斥标记模式，同一时刻最多只能有一个标签段
- `strict interval semantics`：严格区间语义，统一采用 `(tstart, tend]`
- `track edit`：对某一图层中的点位置或元数据做 reviewer 级修正

## 3. Epic 与需求

### Epic A：轨迹编辑模式

#### FR-001：MUST - 系统支持进入“轨迹编辑模式”

验收标准：

- 用户可从轨迹检查区进入显式编辑态。
- 编辑态与普通查看态有明确视觉区分。
- 编辑态关闭后不影响已有 review/timeline 标注读取。

#### FR-002：MUST - 用户可以选中单个点并拖动修改位置

验收标准：

- 支持在可编辑图层中命中单点。
- 拖动结束后生成稳定的点 patch，而不是直接覆写原始 CSV。
- 切换 UID 后可重新读回该修正。

#### FR-003：MUST - 用户可以按住 `Ctrl` 多选点

验收标准：

- `Ctrl + click` 可增量加入/移出选择集。
- 选择集跨拖动、右键菜单和批量修改共用同一状态。
- 非 `Ctrl` 点击会清空旧选择集并切到新的主选择。

#### FR-004：MUST - 用户可以按住 `Shift` 选中一段连续点

验收标准：

- `Shift` 选择以“当前主锚点 + 新点击点”形成连续段。
- 连续段的定义必须基于 canonical point order，而不是临时渲染顺序。

#### FR-005：MUST - 用户可以右键批量修改元数据

验收标准：

- 对选择集中的点支持一次性改标签类字段。
- 批量修改后生成 patch，而不是分别改散落状态。
- 批量修改结果可持久化并回读。

#### FR-006：MUST - 支持多点拖动

验收标准：

- 当选择集包含多个点时，拖动任一选中点会对整组点应用同一位移。
- 位移应用后保持点间相对拓扑不变。

#### FR-007：MUST - 轨迹编辑结果与 timeline 标注、review 结果分离持久化

验收标准：

- `track edit` 有独立 reviewer 级存储。
- 未进入编辑模式时不会污染原始渲染数据。

### Epic B：严格分段语义与导出

#### FR-101：MUST - 时间线支持互斥标记开关

验收标准：

- 用户可打开/关闭互斥模式。
- 打开后，同一时刻最多只有一个标签段有效。
- 关闭后，允许历史兼容读取已有重叠段。

#### FR-102：MUST - 严格逻辑采用 `(tstart, tend]`

验收标准：

- 显示与记录仍使用同样的边界数字。
- 导出与严格判定时，边界点归属于前一段。
- 该语义在 `gps` 与 `signal` 导出上保持一致。

#### FR-103：MUST - 在已有段内部新建段时，起点自动吸附到该段末尾

验收标准：

- 当新建段的起点落入已有段内部时，新段起点自动修正到已有段 `endTime`。
- 若吸附后形成空段，系统拒绝写入。

#### FR-104：MUST - 互斥模式下，新段写入会裁剪或覆盖冲突区间

验收标准：

- 写入后保存结果为 canonical、按时间排序、不可重叠的段列表。
- 旧段被覆盖的部分会被裁剪或移除。

#### FR-105：MUST - `gps` 导出按严格段语义打标签

验收标准：

- 使用 `(tstart, tend]` 进行标签归属。
- 相邻段共享边界时，导出的共享边界点只保留一条，且归属于前一段。

#### FR-106：MUST - `signal` 导出在跨边界时按段切分

验收标准：

- 当一条 `signal/raw` 记录跨越多个标注段时，导出结果按边界切分成多条记录。
- 每条导出记录只对应一个标签。

#### FR-107：SHOULD - 导出 manifest 显式记录 `annotation_source` 与 `interval_semantics`

验收标准：

- reviewer 段、legacy fallback 和区间语义在导出结果中可见。

### Epic C：Studio 导出工作区布局治理

#### FR-201：MUST - Upload / Export 入口切换到对应工作区

验收标准：

- 打开 upload 入口时，左列滚到上传区。
- 打开 export 入口时，左列滚到导出区。
- 切换行为不依赖多级祖先 `scrollIntoView` 的副作用。

#### FR-202：MUST - 响应式断点下只保留明确的 scroll owner

验收标准：

- 桌面态下，左列与右列的滚动职责明确。
- `<=980px` 时不出现 `body + panel + sidebar + list` 多重纵向滚动竞争。

#### FR-203：SHOULD - 折叠/浮层状态有统一的可访问性契约

验收标准：

- trigger 维护 `aria-expanded`
- 内容维护 `aria-hidden`
- 关闭时焦点回到触发器

### 5.2 Track Edit Contract（下一轮落地）

建议结构：

```json
{
  "schema_version": 1,
  "uid": "1001",
  "reviewer_id": "alice",
  "updated_at": "2026-04-27T12:00:00Z",
  "pointPatches": [
    {
      "pointId": "fmm:1001:1710000000000:12",
      "layerKey": "fmm",
      "rowIndex": 12,
      "timestamp": 1710000000000,
      "position": {
        "latitude": 39.9001,
        "longitude": 116.3001
      },
      "metadata": {
        "status": "train"
      }
    }
  ]
}
```

说明：

- `pointId` 必须稳定、可回读、可跨渲染层复用。
- 只保存 patch，不直接复制整层原始数据。

## 6. 关键交互规则

### 6.1 互斥段新建规则

1. 用户提出新段 `[start, end]`
2. 若 `start` 落入已有段内部，则将 `start` 吸附到该段 `end`
3. 若吸附后 `start >= end`，拒绝创建
4. 用新段覆盖冲突区间
5. 输出 canonical、不重叠的段列表

### 6.2 严格边界规则

- 记录和显示仍使用同样的边界数值。
- 严格逻辑按 `(tstart, tend]` 判定。
- 相邻两段 `[A.start, A.end]` 与 `[A.end, B.end]` 共享边界时：
  - 共享边界属于前一段
  - 后一段从该边界之后开始生效

### 6.3 Track Edit 交互规则

- 单击：选择单点
- `Ctrl + 单击`：增量多选
- `Shift + 单击`：按 canonical order 形成连续段选择
- 拖动：
  - 单点选择时移动单点
  - 多点选择时整体平移
- 右键：
  - 对当前选择集打开批量元数据菜单

## 8. 风险与取舍

- 本轮不一次做完整编辑模式，是为了避免把点编辑直接塞回 `track_inspection.js` 热点。
- 互斥段语义先做“规范化 + 导出正确性”，比先补更多入口更值。
- Studio 导出工作区先修滚动和布局契约，比直接改成整页路由风险更低。
