# trajectory_annotation_studio 当前落成情况

## 1. 当前定位

`trajectory_annotation_studio` 目前已经不是单纯挂在 `cellular_quality` 里的临时页面，而是一个可以独立启动、独立承载批次数据、独立写回审核结果的本地标注工作台。

当前最适合的使用方式是：

- 作为 `organizer / reviewer` 的审阅与标注工具
- 面向批次目录 (`batches-root`) 工作
- 允许不同来源的数据以统一批次格式接入
- 把审核结果和时间轴标注结果跟随批次一起保存

它现在已经能支撑“组内共享一套服务 + 多人各自准备数据批次”的工作流，但还不等于完整的平台化系统。

## 2. 已经落成的能力

### 2.1 批次发现与切换

当前服务支持两种运行方式：

1. 单批次直连模式  
   直接通过 `--result-root --review-root --export-root` 指到一批结果。
2. 批次根目录模式  
   通过 `--batches-root` 指向一个批次根，服务会自动发现所有子批次，并在前端支持批次切换。

对于组内共享，推荐固定使用第 2 种。

### 2.2 轨迹审阅主流程

前端已支持：

- 三列审阅箱：`待审 / 已通过 / 未通过或跳过`
- 轨迹搜索
- 基于状态标签的筛选
- 点选任意 UID/轨迹进入当前审阅
- 已审结果重新打开并编辑
- 保存后自动流转到下一条待审

### 2.3 地图与图层展示

当前页面支持两类 UI 模式：

1. `chain2`
   默认模式，面向 `raw / snap / od / fmm / line`
2. `sim_signal`
   面向 `gps / signal / stations`

当前地图能力包括：

- 多图层叠加显示
- 图层显隐、颜色、透明度控制
- 标签直接显示开关
- 标签随缩放变化
- 时间、停留时长、基站聚合信息展示
- 大批量点位下的采样与 bucket 渲染优化

### 2.4 时间轴与标注

当前已经落成：

- 日期区间选择
- 时间轴窗口与全时间总览条
- 拖动式时间光标
- 光标对应点信息浮窗
- 右键菜单
- 单点图钉标记
- 段落标记
- 段落预览跟随鼠标
- 段落删除
- 段落重叠最多 3 条 lane，更多重叠按 cover 逻辑展示

### 2.5 审核与导出

当前已经支持：

- 整轨审核写回：`accept / reject / skip`
- `reference_source` 指向 `line.csv` 或 `fmm.csv`
- `accept` 时校验参考文件必须真实存在且非空
- `accepted_assets` 导出
- 时间轴标记写回

## 3. 当前真实数据落盘位置

以下是当前实现里的真实写回位置，不是计划中的接口：

### 3.1 整轨审核

位于批次目录下：

```text
<batch_root>/
  review/
    ledger.jsonl
    latest_reviews.json
```

说明：

- `ledger.jsonl` 是 append-only 审核流水
- `latest_reviews.json` 是按 UID 聚合后的最新视图

### 3.2 时间轴标记

位于批次目录下：

```text
<batch_root>/
  review/
    timeline_annotations/
      <uid>.json
      ledger.jsonl
```

说明：

- `<uid>.json` 是当前 UID 最新标注快照
- `timeline_annotations/ledger.jsonl` 是时间轴标注流水

### 3.3 accepted 导出

位于批次目录下：

```text
<batch_root>/
  accepted_assets/
    samples/
      <sample_id>/
        raw.csv
        line.csv | fmm.csv
        review.json
    accepted_reviews.jsonl
    export_manifest.json
```

## 4. 当前仍然没有落成的部分

下面这些不要误判成“已经可用”：

### 4.1 还没有鉴权

当前 `review_server.py` 是 stdlib HTTP 服务，不带：

- 登录
- 用户权限
- 并发冲突处理
- 审批流

如果要放到组内服务器上，鉴权要由外层反向代理或内网环境负责。

### 4.2 还没有中心化标签目录

时间轴上的“自定义类别”目前主要保存在浏览器 `localStorage`，不是服务端统一配置。

这意味着：

- 已保存的段落标记会跟批次一起保存，因为服务端写回了 `categoryId / categoryName / color`
- 但“右键菜单里有哪些可新建标签”目前仍然由当前浏览器本地设置决定

所以当前版本适合先统一批次格式和落盘位置，不适合宣称已经完成“多人统一标签治理”。

### 4.3 `segments/` 目录不是当前主存储

部分旧批次目录里仍然有：

```text
<batch_root>/segments/
```

这是早期遗留结构。当前页面实际使用的是：

```text
<batch_root>/review/timeline_annotations/
```

因此后续新批次不应再把 `segments/` 作为主契约。

## 5. 适合进入组内共享的边界

当前版本适合推进到：

- 组内共享一套服务
- 大家按统一批次格式接入自己的数据
- organizer / reviewer 在统一页面里审阅
- 已审核结果与时间轴标注随批次目录持久化保存

当前版本不适合直接承诺：

- 公网暴露
- 多团队正式平台
- 强一致多人协同
- 服务端统一标签治理

## 6. 推荐下一步

如果要把它放到组内服务器上，建议按下面顺序推进：

1. 先冻结批次目录合同、CSV 合同、输出位置
2. 再固定服务器上的 `batches-root`
3. 再决定是否加鉴权
4. 最后再推进统一标签目录与多人协作规则
