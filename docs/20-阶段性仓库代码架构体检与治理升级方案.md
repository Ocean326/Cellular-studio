## 1. 背景与目标

这份文档用于给 `trajectory_annotation_studio` 做一版阶段性体检，不只回答“现在哪里不健康”，也把后续应该长期执行的治理机制一起固定下来。

当前判断基于以下事实：

- 仓库已经不是一次性 demo，而是承载真实批次、真实 review 写回、多人 reviewer namespace、时间轴标注和导出链路的工作台。
- 本地核心单测当前可通过：
  `python3 -m unittest trajectory_annotation_studio.web.test_review_lib trajectory_annotation_studio.web.test_review_server trajectory_annotation_studio.scripts.tests.test_server_batch_tools trajectory_annotation_studio.scripts.tests.test_research_arena_v15_layer_adapter`
- 当前主运行面由以下模块承载：
  - [`../web/index.html`](../web/index.html)
  - [`../web/review_server.py`](../web/review_server.py)
  - [`../web/review_lib.py`](../web/review_lib.py)
  - [`../scripts/server_batch_lib.py`](../scripts/server_batch_lib.py)
  - [`../scripts/start_review_studio.py`](../scripts/start_review_studio.py)

这轮文档采用两层方法：

- `architect`
  - 主 lane：`codebase-architecture`
  - 副 lane：`architecture-governance`
- `thinking-lenses`
  - 模式：`design`
  - delivery mode：`inline-structured`
  - 主 lens：`First principles`
  - 辅助 lens：`MECE`

也就是先把问题拆成“仓库、代码、架构、治理”四层，再给每层定义可执行、可验证、可持续维护的升级方法。

## 2. 当前架构快照

### 2.1 当前实际分层

当前系统可以粗分为 4 层：

1. 页面与交互层
   - [`../web/index.html`](../web/index.html)
   - 承担 UI、状态管理、地图渲染、时间轴、review 表单、批次切换、annotation settings
2. HTTP 入口层
   - [`../web/review_server.py`](../web/review_server.py)
   - 承担路由、批次发现、静态文件服务、导出入口、upload 入口
3. review / export / aggregate 领域逻辑层
   - [`../web/review_lib.py`](../web/review_lib.py)
   - 承担 reviewer namespace、review 写回、timeline annotations、aggregate、export
4. 批次发布与校验脚本层
   - [`../scripts/server_batch_lib.py`](../scripts/server_batch_lib.py)
   - [`../scripts/publish_server_batch.py`](../scripts/publish_server_batch.py)
   - [`../scripts/validate_published_batch.py`](../scripts/validate_published_batch.py)

### 2.2 当前健康信号

正向信号：

- review / timeline / export 语义已经从页面中抽到了 [`../web/review_lib.py`](../web/review_lib.py)，不是所有逻辑都堆在 HTTP 层。
- 批次目录、reviewer namespace、aggregate、export 已经有清晰的持久化模型，见 [README.md](../README.md) 当前存储结构说明。
- 有最小但真实的 unittest 覆盖，并且覆盖了 reviewer namespace、aggregate、timeline annotations、published batch 校验等关键链路。
- 部署、接入、179 pilot、发布与校验文档已经比较完整，说明仓库有治理意识，不是“纯代码无合同”状态。

风险信号：

- 前端交互壳体仍集中在一个超大单文件里，`index.html` 已达 6000+ 行。
- 包边界尚未稳定，模块导入仍依赖脚本相对路径而不是稳定包名。
- manifest / layer_specs / review_reference 合同在运行层和脚本层双份实现。
- 仓库存在历史快照与运行产物混杂，主干结构不够收口。
- 有本地单测，但没有仓库级 CI 护栏。

## 3. 阶段性体检结论

### 3.1 总体结论

`trajectory_annotation_studio` 当前处于：

- 产品可用性：`可持续迭代`
- 架构成熟度：`从原型走向工具平台的中间阶段`
- 代码健康度：`可维护，但高耦合热点已经形成`
- 仓库治理度：`有意识，但机制还没有完全落地`

不建议把它定义为“需要重写”的状态。

更准确的判断是：

- 它已经足够好，值得继续演进
- 但再继续按当前结构加功能，会越来越依赖个别维护者的脑内上下文
- 现在最需要的是“收边界、补机制、控热点”，不是换框架

### 3.2 分维度体检

| 维度 | 当前判断 | 主要证据 | 风险级别 |
| --- | --- | --- | --- |
| 仓库健康 | 中等偏弱 | 主干里仍保留 `_backup_*`，`.runtime/` 未忽略，存在旧 README 口径 | 高 |
| 代码健康 | 中等 | 核心逻辑有抽层，但前端单文件过大，部分合同解析重复 | 高 |
| 架构健康 | 中等 | review 领域模型清楚，但 transport / contract / frontend boundary 还未完全固化 | 高 |
| 测试健康 | 中等偏好 | 本地 21 个 unittest 通过，但没有 CI，也缺浏览器 smoke | 中 |
| 文档健康 | 较好 | 文档体系完整，但存在旧入口与新入口并存 | 中 |
| 接入治理 | 中等偏弱 | 原则要求走 `adapters/`，但真实 adapter 形态还未成型 | 中 |

## 4. 关键问题与风险分级

### 4.1 P1：必须优先治理的问题

#### P1-1. 包边界未固化

当前 [`../web/review_server.py`](../web/review_server.py) 直接使用：

- `from review_lib import ...`

而测试也在手动修改 `sys.path` 来保证导入成功：

- [`../web/test_review_server.py`](../web/test_review_server.py)
- [`../web/test_review_lib.py`](../web/test_review_lib.py)

这意味着：

- 脚本位置就是运行约束
- `python -m trajectory_annotation_studio.web.review_server` 这类包方式目前不稳定
- 代码“能跑”依赖调用姿势，而不是正式模块边界

这类问题短期不一定阻塞功能，但会长期拉高：

- 测试门槛
- CLI 稳定性
- 后续 CI / packaging / release 脚本一致性成本

#### P1-2. 前端热点过大

[`../web/index.html`](../web/index.html) 同时承载：

- HTML 结构
- 样式
- UI mode preset
- 数据加载
- 批次切换
- reviewer session
- triage board
- 地图渲染
- 时间轴
- annotation settings
- export / review 表单交互

这不是“写法不好看”的问题，而是架构问题：

- 任何一个新需求都在修改高共享壳体
- 功能回归面很大
- 很难给子模块单独建立验证边界
- 新接手的人需要先理解整页 global state

#### P1-3. 批次合同存在双份实现

以下两处都在解析 manifest / layer_specs / review_reference：

- [`../web/review_lib.py`](../web/review_lib.py)
- [`../scripts/server_batch_lib.py`](../scripts/server_batch_lib.py)

这会产生一个长期隐患：

- 发布脚本认为“合法”的批次
- 运行时真正能正确消费的批次
- 导出时真正保留的 review reference 文件

三者可能逐渐漂移。

对于这个仓库来说，批次合同不是普通工具函数，而是平台级共享内核，不能允许多份实现长期并存。

### 4.2 P2：如果继续放着，会持续放大维护成本的问题

#### P2-1. 仓库主干事实源还不够单一

当前主仓里仍有：

- `_backup_web_20260410_222758/`
- `_backup_new_stack_20260410_223335/`

同时 [`../web/README.md`](../web/README.md) 还保留旧的 `cellular_quality/src/web` 启动口径。

这会产生“知识分叉”：

- 哪份才是当前真实主干
- 哪些说明还能信
- 哪些目录只是历史参考

仓库层面的健康问题，长期会直接转化为：

- 新人接手成本
- 错误修改旧目录
- 文档不再可信

#### P2-2. 适配层原则已经写下，但还没产品化

[`../CONTRIBUTING.md`](../CONTRIBUTING.md) 和 [`../adapters/README.md`](../adapters/README.md) 都明确说：

- 数据差异优先收敛到 `adapters/`

但现实里：

- `adapters/` 只有说明，没有真实 adapter 样板
- 真正的数据接入脚本仍主要散落在 `scripts/`
- 个别脚本还携带明显环境耦合和私有路径假设

这说明治理原则正确，但组织形态还没闭环。

#### P2-3. CI 缺位

当前能本地跑 unittest 是优点，但仓库里没有 `.github/workflows/`，意味着：

- 当前健康更多依赖维护者自觉
- 包导入退化、README 漂移、脚本回归都可能在 merge 后才暴露

### 4.3 P3：现在先记录，不建议马上重做的问题

- 不建议立即重写成 React/Vite/FastAPI 新栈
- 不建议当前就引入数据库替代 review 文件落盘
- 不建议现在就做复杂权限系统
- 不建议现在就把 `review_server.py` 强行拆成多进程并发服务

原因不是这些方向永远不做，而是当前最主要的问题并不是“技术选型太旧”，而是：

- 模块边界不够稳
- 共享合同没有唯一实现
- 仓库治理护栏还没补齐

在这些问题未收敛之前，换栈只会把问题搬进新壳。

## 5. 推荐的目标边界模型

### 5.1 四层目标模型

建议把后续主干固定为四层：

1. `presentation`
   - 页面壳体、交互编排、UI 状态
2. `application`
   - batch catalog、review session、timeline orchestration、export trigger
3. `domain`
   - review record、timeline annotation、review aggregate、batch contract
4. `infrastructure`
   - HTTP handler、filesystem persistence、zip upload、batch publish/validate

对于当前仓库，最关键的不是把它拆成很多目录，而是先把“谁能依赖谁”固定：

- `presentation` 不直接解析底层文件合同
- `review_server.py` 不重复发明 review / aggregate 规则
- `scripts/` 不重新实现运行时合同

### 5.2 共享内核

后续应明确以下两类共享内核：

1. `batch contract kernel`
   - manifest 读取
   - layer_specs 归一化
   - review_reference 语义
   - export 文件名语义
2. `review domain kernel`
   - reviewer id / profile 归一化
   - review payload 校验
   - timeline annotation 校验
   - aggregate 生成

其中第 1 类目前分散，第 2 类已基本沉到 [`../web/review_lib.py`](../web/review_lib.py)。

## 6. 机制设计

这部分不只是“建议”，而是未来应长期执行的健康机制。

### 6.1 仓库健康机制

#### 机制 A：主干收口机制

规则：

- `main` 只保留当前真实运行路径
- 历史快照不长期驻留主干
- 运行产物、日志、pid、supervisor 输出必须进入忽略规则

最小动作：

- 把 `.runtime/` 加入 `.gitignore`
- 将 `_backup_*` 从主干移出，转为：
  - archive 分支
  - release tarball
  - 明确标记为 `docs/archive/` 的设计记录
- 修正 [`../web/README.md`](../web/README.md) 中的旧路径与旧仓名

验收口径：

- 新同学进入仓库后，不会误把备份目录当主干
- 根 README、web README、docs 索引对运行入口描述一致

#### 机制 B：文档可信度机制

规则：

- 凡是入口、部署命令、目录合同发生变化，必须同步更新：
  - [`../README.md`](../README.md)
  - [`./README.md`](./README.md)
  - 对应专项文档

建议做法：

- 每次 PR 模板增加一项：
  - `是否影响入口/合同/部署文档`

### 6.2 代码健康机制

#### 机制 C：包导入稳定机制

目标：

- 所有核心模块既能脚本运行，也能包方式运行

要求：

- `web/` 与 `scripts/` 中共享逻辑不再依赖裸相对导入
- 测试不再通过手动修改 `sys.path` 兜底作为常态
- `python -m trajectory_annotation_studio.web.review_server` 应可直接运行

建议做法：

- 给 `trajectory_annotation_studio`、`web`、`scripts` 明确 package 边界
- 将跨模块导入切成包名导入
- 为 CLI 保留轻量入口脚本，但入口只做参数解析和调度

#### 机制 D：热点文件预算机制

目标：

- 控制“任何功能都得改同一个巨型文件”的趋势

建议预算：

- 页面壳体文件尽量不再继续显著膨胀
- 新增能力优先拆到：
  - `timeline`
  - `review-session`
  - `batch-catalog`
  - `map-render`
  - `annotation-settings`

这里的关键不是立刻换框架，而是先做“功能切片”。

#### 机制 E：共享合同唯一实现机制

目标：

- manifest / layer_specs / review_reference 语义只有一份权威实现

要求：

- 运行时与脚本层共同依赖同一个 contract 模块
- 任何批次合同变更必须加 contract test

验收口径：

- published batch validate 通过的批次，运行时必须能消费
- runtime/export/validate 三处对 review_reference 的判断一致

### 6.3 架构治理机制

#### 机制 F：ADR 轻量化机制

从现在开始，以下改动需要有 ADR 或 ADR backlog：

- 是否继续维持 stdlib server
- 前端是否继续原生模块化，还是迁到正式前端框架
- batch contract 模块的最终位置
- adapters 与 scripts 的职责边界

建议不必一次写很重，最少采用：

- Decision
- Context
- Options
- Recommendation
- Consequences

#### 机制 G：合同变更门禁

以下改动必须触发专项检查：

- `manifest.json` 字段语义变化
- review 存储结构变化
- export 目录语义变化
- reviewer namespace 兼容变化

门禁内容：

- 文档更新
- unittest 更新
- 兼容/迁移说明
- rollback 方法

### 6.4 验证机制

#### 机制 H：仓库级 CI

最小 CI 不需要复杂，但必须存在：

1. Python import smoke
   - `python -m trajectory_annotation_studio.web.review_server --help`
2. 核心 unittest
3. 前端脚本 parse smoke
4. 文档链接或 README 基本检查

#### 机制 I：发布前 smoke

在本地或服务器发布前，固定检查：

1. `/api/health`
2. `/api/batches`
3. reviewer session
4. write review
5. write timeline annotation
6. export accepted / reviewer bundle

这部分已经在测试里覆盖了部分逻辑，但需要一个更贴近 operator 的发布 smoke 脚本。

## 7. 分阶段升级路线

### Phase 0：仓库收口与护栏补齐

目标：

- 先把仓库事实源、入口和 CI 收稳

交付物：

- `.runtime/` ignore
- 清理或迁移 `_backup_*`
- 修复旧 README 入口
- 增加最小 CI
- 增加仓库健康 checklist

完成标志：

- 主干不再混杂旧主线与运行产物
- 新同学按 README 能找到唯一启动路径

### Phase 1：包边界与共享合同收敛

目标：

- 让模块以“包边界”而不是“脚本位置”稳定运行

交付物：

- 稳定包导入
- shared batch contract 模块
- 对 manifest/review_reference/export contract 的专项测试

完成标志：

- `python -m ...review_server` 可运行
- review_lib 与 server_batch_lib 不再双份维护同一合同语义

### Phase 2：前端热点切片

目标：

- 在不换主栈的前提下降低 `index.html` 的耦合

交付物：

- `index.html` 只保留页面壳与启动逻辑
- 时间轴、review session、annotation settings、batch catalog、map render 切为独立模块
- 前端 smoke 与关键交互回归用例

完成标志：

- 单个功能改动无需默认触碰整页 global state

### Phase 3：适配层产品化

目标：

- 让“adapters 优先”从原则变成真实工作流

交付物：

- 至少一个真实 adapter 样板
- `adapters/<dataset>/build_batch.py` 约定
- adapter contract test

完成标志：

- 接新数据时优先复制 adapter 模板，而不是直接改核心脚本

### Phase 4：运行与发布硬化

目标：

- 让 operator 与 maintainer 的工作流更稳定可重复

交付物：

- release smoke script
- operator checklist
- published batch verify + review smoke 串联脚本

完成标志：

- 新 operator 可以按文档和脚本完成一次完整发布与回归

## 8. 近期建议执行顺序

当前最推荐的顺序是：

1. 先做 `Phase 0`
2. 再做 `Phase 1`
3. 之后再决定 `Phase 2` 的切片方式

不要反过来先上大规模前端重构。

如果 `Phase 0` 和 `Phase 1` 没做，后续无论换 React、换 FastAPI、换数据库，都会把当前的问题带进下一层。

## 9. ADR Backlog 建议

建议立即建立以下 ADR backlog：

1. `ADR-001`：`trajectory_annotation_studio` 的正式包边界与运行入口
2. `ADR-002`：批次合同共享内核的位置与职责
3. `ADR-003`：前端模块化策略
   - 继续原生 ES modules
   - 还是迁到正式前端框架
4. `ADR-004`：`scripts/` 与 `adapters/` 的职责分界
5. `ADR-005`：发布前 smoke 与 CI 的最小必选项

## 10. 阶段性验收口径

这份文档不是为了追求“写完一个大方案”，而是为了给后续 1 到 2 个版本的治理提供验收标准。

如果下一阶段算“明显更健康”，至少应满足：

1. 仓库主干只有一套真实入口和一套真实说明
2. 包导入不再依赖手工 `sys.path` 兜底
3. batch contract 只有一份权威实现
4. 有最小 CI 护栏
5. 新数据接入可以按 adapter 模板走，而不是默认去改核心壳体

## 11. 当前建议

当前最优方向不是“推翻重来”，而是：

- 保留现有 review / export / reviewer namespace 这条已经跑通的核心主线
- 优先修复包边界、共享合同和仓库治理
