# 长期可维护性整改与轨迹标注平台升级总方案

更新时间：`2026-04-27`

## 0. 路由结论

### `architect-router`

- 架构问题形态：这是一次跨越代码治理、前后端边界、持久化合同、复杂交互模型和导出语义的综合升级，不是单点 bugfix。
- 选定 lane：`architect`
- 为什么是这个 lane：
  - 需要同时处理 codebase refactor、规格落地和后续多轮功能演进。
  - “轨迹编辑模式”会直接改变前端状态模型、后端存储合同和导出语义，已经超过单一 `codebase-architecture` 的范围。
- 考虑过但未采用的邻近 lane：
  - `codebase-architecture`：只够回答“怎么拆模块”，不够覆盖功能升级的长期治理。
  - `spec-driven-architecture`：规格很重要，但当前 blocker 不只是规格不清，还有现有实现热点过大、状态无专属模型。
- 下一步具体动作：先冻结整改边界、代码规范和分阶段切片，再在首轮切片上实施。

### `delivery-router`

- 任务类型：端到端交付型升级，请求同时包含长期治理、需求规格、实现、测试和文档。
- 选定 lane：`delivery-conductor`
- 为什么不是只走 `product-manager`：
  - `product-manager` 适合把需求写清楚，但这轮还必须落到代码与验证。
  - 当前真实 blocker 是“要把多轮升级做成持续可交付的工程路径”，不只是“写一份 PRD”。
- 下一步具体动作：由主线程充当本轮 delivery owner，先产出整改方案和 tech spec，再选择首轮最值得动的切片实施。
- 是否保持单 lane：否。本轮以 `delivery-conductor` 方式推进，但显式使用 `product-manager` 产出可测试规格。

## 1. 现状体检结论

当前平台已经从 demo 进入“真实多人使用的工作台”阶段，优点是：

- reviewer namespace、aggregate、导出链路已经成型。
- `index.html` 已经从超大单文件拆成 `runtime + feature` 目录。
- Python 侧已有不少关键链路测试。

但从“长期可维护 + 反复加功能”的角度，当前短板已经很明确。

### P1：必须先补的短板

#### P1-1. 缺少“可编辑轨迹”的一等领域模型

现在系统里真正被持久化的标注对象只有：

- 整轨 review
- timeline pins
- timeline segments

还没有“轨迹点编辑”这个独立对象，因此以下能力都没有落脚点：

- 点选择集
- 多点拖动
- 点元数据批量修改
- 图层内点引用 ID
- 轨迹编辑的 reviewer 级持久化
- 编辑命令与撤销边界

这意味着如果直接把拖点、多选、右键改元数据塞进现有 `track_inspection.js`，会继续堆在 UI 热点里。

#### P1-2. 时间段当前只是“能画出来”，不是“有严格语义的区间系统”

目前 timeline segments 具备基础可用性，但还缺三种平台级语义：

- 互斥约束：同一时刻只能归属于一个标签
- 边界归属：导出和严格逻辑要统一成 `(tstart, tend]`
- 新建覆盖策略：在已有段上新建段时，如何吸附、裁剪和覆盖

不补这层语义，导出结果会依赖“段顺序副作用”和“first-match 规则”，而不是显式规则。

#### P1-3. 状态虽已拆文件，但所有权还不够明确

当前 `web/app/runtime/state.js` 仍持有大量跨域全局可变状态，`track_inspection.js` 仍然承担：

- timeline 交互
- 地图渲染
- 日期窗口
- annotation settings
- 局部持久化
- 服务端同步

这对“想到哪里加哪里”的开发方式非常不友好，因为任何功能都容易越过边界直接改全局状态。

#### P1-4. 导出链路还没有“合同优先”的测试口径

已有测试覆盖了不少 happy path，但这轮新增能力最关键的边界仍未被护栏化：

- 重叠段如何处理
- 共享边界归属谁
- signal 跨边界记录如何切分
- export workspace 在不同 viewport 下的滚动所有权
- 折叠/浮层状态和 `aria` 契约

### P2：持续放大会拖慢后续迭代的短板

#### P2-1. 复杂交互还没有专门的 command model

轨迹编辑模式一旦进入：

- 点选中
- 框选 / 多选
- 拖动
- 批量修改
- 段编辑

这些都不应直接通过“某个事件里就地改数组”实现，而应该有显式命令和归一化入口。

#### P2-2. 前端导出工作区仍是“伪页面”

现在导出区并不是独立页面或独立 workspace，而是 Studio modal 左列滚动区里的第二段内容。两个浮动入口只是：

- 打开同一个 overlay
- 滚动到不同位置

这正是当前“导出页 / 折叠卡片 / 浮层重叠”问题的根因。

#### P2-3. 新增功能文档入口还不够集中

目前文档体系很完整，但这次升级涉及：

- 轨迹编辑模式
- 严格段语义
- 导出接口
- 布局治理

如果没有单独的总方案和 tech spec，后续容易再次散落到聊天、patch 和记忆里。

## 2. 这轮之后必须执行的代码规范

为了适应“经常想到哪里加哪里”的迭代方式，后续新增代码按下面规则执行。

### 2.1 前端规范

1. 新交互必须先有“状态归属”，再写事件处理。
2. `runtime/state.js` 只允许持有页面级上下文和已声明 owner 的共享状态。
3. 复杂 feature 必须至少拆成 4 类职责：
   - `*_contract`
   - `*_state`
   - `*_actions` 或 `*_controller`
   - `*_view`
4. 视图层不直接发明业务规则；业务规则只能写在 feature contract / action 或 Python domain 层。
5. 任何新持久化字段都要有默认值、normalize 逻辑和回读兼容。
6. 任何涉及多选、拖拽、右键菜单的功能，都必须先定义命令模型和冲突策略。

### 2.2 后端规范

1. `review_server.py` 只做 transport、参数校验和分发，不新增领域规则。
2. review / annotation / export 的领域规则只允许进入 `review_lib.py` 或新的 domain module。
3. 任何导出合同变更，都要先写文档，再补测试，再改实现。
4. 新 schema 必须支持旧数据回读，不能依赖“人工先迁完再运行”。

### 2.3 测试规范

1. 新增交互至少覆盖：
   - 数据 normalize
   - 状态变换
   - 服务端持久化
   - 导出结果
2. 新增布局至少覆盖：
   - DOM 存在
   - 状态契约
   - 响应式滚动 owner
3. 凡是“边界归属”“互斥规则”“自动裁剪/吸附”一类逻辑，都必须有明确边界用例。

### 2.4 文档规范

1. 新功能先写 tech spec，再开始大改。
2. 新持久化结构要落到 docs 中，不能只藏在测试里。
3. 每一轮只允许一个“活文档”描述当前升级状态，避免多个 patch memo 并行漂移。

## 3. 目标架构边界

### 3.1 前端

建议把后续主边界固定成 4 层：

1. `shell`
   - modal 打开关闭
   - workspace 切换
   - 全局快捷键
2. `workspace`
   - batch / uid / manifest / layer snapshot
3. `review_flow`
   - reviewer session
   - review decision
   - aggregate
4. `track_inspection`
   - timeline segment policy
   - time scrubber
   - map interaction
   - track edit session

在 `track_inspection` 内，再拆出两条一等子域：

- `timeline_annotation`
- `trajectory_edit`

### 3.2 后端

建议把 Python 侧分成 3 个清晰内核：

1. review domain
   - review record
   - reviewer aggregate
2. annotation domain
   - pins
   - segments
   - segment policy
3. track edit domain
   - point patch
   - metadata patch
   - edit ledger

### 3.3 数据合同

这轮之后要明确三类合同：

1. timeline annotation contract
   - `segments`
   - `pins`
   - `segmentPolicy`
2. track edit contract
   - 点引用
   - patch 表达
   - reviewer 级持久化
3. export contract
   - 边界语义
   - annotation source
   - signal/gps 的严格规则

## 4. 分阶段整改与升级计划

### Phase 0：护栏固化

目标：

- 先写总方案、tech spec、测试清单
- 给时间段严格语义和导出规则定口径

验收：

- 新文档落库
- 首轮实现范围明确

### Phase 1：严格段语义与导出链路

目标：

- 给 timeline segments 增加 `segmentPolicy`
- 支持互斥模式
- 导出统一 `(tstart, tend]`
- signal 跨边界按段切分

验收：

- 重叠段不再无约束持久化
- dataset export 不再依赖 first-match 副作用
- 测试覆盖边界归属、legacy fallback、signal 切分

### Phase 2：Studio 导出工作区布局治理

目标：

- upload / export workspace 切换从“伪页面滚动”变成显式契约
- 收敛滚动 owner
- 避免导出区、help/custom-fields 和折叠卡片相互挤压

验收：

- 打开 export 必达导出区
- 打开 upload 必达上传区
- `<=980px` 只有一个主要纵向滚动 owner

### Phase 3：轨迹编辑模式基础设施

目标：

- 定义 canonical point ref
- 定义 reviewer 级 `track_edits`
- 定义选择集、批量 patch 和命令模型

验收：

- 能稳定引用点，而不是直接靠 row index 和临时变量驱动
- 编辑持久化合同落库并有兼容回读

### Phase 4：完整轨迹编辑模式

目标：

- 点拖动
- Ctrl 多选点
- Shift 选段
- 右键批量改元数据
- 多点拖动

验收：

- 地图侧和持久化侧共用一套点引用和 patch 合同
- 交互结果可导出、可回读、可测试

## 5. 本轮实施范围

本轮不强行一次做完整轨迹编辑模式，而是按“先补地基，再落最值钱的功能”推进。

本轮实际实施面：

1. 文档化整改方案、功能规格和测试清单
2. 实现 timeline `segmentPolicy`
3. 实现互斥模式下的严格段语义与导出规则
4. 修复 Studio upload/export workspace 的滚动与布局热点
5. 为轨迹编辑模式写清楚数据模型和下一轮切口

本轮明确不做：

- 完整多点编辑 UI
- 撤销/重做
- 点新增 / 点删除
- 大规模重写前端框架

## 6. 下一轮的首要工程目标

下一轮优先不是“继续堆更多按钮”，而是：

- 把 `trajectory_edit` 从 `track_inspection.js` 里独立成子域
- 给点编辑建立独立存储合同
- 在有 contract 的前提下再上拖点、多选和批量改元数据

只有这样，后面你继续“想到哪里加哪里”，系统才不会再次回到无边界增长。
