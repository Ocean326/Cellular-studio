# trajectory_annotation_studio 当前落成情况

## 1. 当前定位

`trajectory_annotation_studio` 目前已经不是单纯挂在 `cellular_quality` 里的临时页面，而是一个可以独立启动、独立承载批次数据、独立写回审核结果的本地标注工作台。

当前最适合的使用方式是：

- 作为 `organizer / reviewer` 的审阅与标注工具
- 面向批次目录 (`batches-root`) 工作
- 允许不同来源的数据以统一批次格式接入
- 把审核结果和时间轴标注结果跟随批次一起保存

它现在已经能支撑“组内共享一套服务 + 多人各自准备数据批次”的工作流，但还不等于完整的平台化系统。

## 2. 已经落成的能力

### 2.1 批次发现与切换

当前服务支持两种运行方式：

1. 单批次直连模式  
   直接通过 `--result-root --review-root --export-root` 指到一批结果。
2. 批次根目录模式  
   通过 `--batches-root` 指向一个批次根，服务会自动发现所有子批次，并在前端支持批次切换。

对于组内共享，推荐固定使用第 2 种。

### 2.2 轨迹审阅主流程

前端已支持：

- 三列审阅箱：`待审 / 已通过 / 未通过或跳过`
- 轨迹搜索
- 基于状态标签的筛选
- 点选任意 UID/轨迹进入当前审阅
- 已审结果重新打开并编辑
- 保存后自动流转到下一条待审

### 2.3 地图与图层展示

当前页面支持两类 UI 模式：

1. `chain2`
   默认模式，面向 `raw / snap / od / fmm / line`
2. `sim_signal`
   面向 `gps / signal / stations`

当前地图能力包括：

- 多图层叠加显示
- 图层显隐、颜色、透明度控制
- 标签直接显示开关
- 标签随缩放变化
- 时间、停留时长、基站聚合信息展示
- 大批量点位下的采样与 bucket 渲染优化

### 2.4 时间轴与标注

当前已经落成：

- 日期区间选择
- 时间轴窗口与全时间总览条
- 拖动式时间光标
- 光标对应点信息浮窗
- 右键菜单
- 单点图钉标记
- 段落标记
- 段落预览跟随鼠标
- 段落删除
- 段落重叠最多 3 条 lane，更多重叠按 cover 逻辑展示

### 2.5 审核与导出

当前已经支持：

- 整轨审核写回：`accept / reject / skip`
- reviewer session：首次进入要求姓名/昵称，支持保持与切换当前标注者
- reviewer 命名空间：整轨审核和时间轴标记按 reviewer 分目录保存
- 多人汇总：同一 UID 可查看多个 reviewer 的最新审核摘要
- `reference_source` 指向 `line.csv` 或 `fmm.csv`
- `accept` 时校验参考文件必须真实存在且非空
- `accepted_assets` 按 reviewer 导出
- `reviewer bundle export`：把“我的审核 + 时间轴标记 + 对应原始工件”打成稳定目录或 zip
- `review aggregate` 汇总导出
- 时间轴标记写回

### 2.6 面向 179 的共享运行基础能力

当前已经具备第一版 server-facing 能力：

- 179 目录初始化脚本
- `published batch + source_result_root` 挂载式批次发布
- 已发布批次结构校验
- Web 上传 zip 到 `incoming`
- `incoming` 上传记录列表
- `incoming` zip 解包、识别标准结果根、校验并按需发布的 CLI
- `systemd` 与 `nginx` 配置模板

这些能力的边界是：

- 已经足够支撑 `private pilot` 的 operator 工作流
- 还不等于“179 已正式上线并稳定服务”

## 3. 当前真实数据落盘位置

以下是当前实现里的真实写回位置，不是计划中的接口：

### 3.1 整轨审核

位于批次目录下：

```text
<batch_root>/
  review/
    system/
      reviewer_registry.json
    reviewers/
      <reviewer_id>/
        profile.json
        reviews/
          ledger.jsonl
          latest_reviews.json
    aggregate/
      stats.json
      by_uid/
        <uid>.json
```

说明：

- 当前主存储不再是单个全局 `ledger.jsonl + latest_reviews.json`
- 每个 reviewer 都有自己的审核流水和 latest 视图
- `aggregate/by_uid/<uid>.json` 用来汇总多人对同一轨迹的最新结果

### 3.2 时间轴标记

位于批次目录下：

```text
<batch_root>/
  review/
    reviewers/
      <reviewer_id>/
        timeline_annotations/
          <uid>.json
          ledger.jsonl
```

说明：

- `<uid>.json` 是某个 reviewer 对当前 UID 的最新标注快照
- `timeline_annotations/ledger.jsonl` 是该 reviewer 自己的时间轴标注流水

### 3.3 accepted 导出

位于批次目录下：

```text
<batch_root>/
  accepted_assets/
    reviewers/
      <reviewer_id>/
        samples/
          <sample_id>/
            raw.csv
            line.csv | fmm.csv
            review.json
        accepted_reviews.jsonl
        export_manifest.json
  review_exports/
    aggregate/
      by_uid.jsonl
      by_reviewer.jsonl
      conflicts.jsonl
      export_manifest.json
```

## 4. 当前仍然没有落成的部分

下面这些不要误判成“已经可用”：

### 4.1 还没有鉴权

当前 `review_server.py` 是 stdlib HTTP 服务，不带：

- 登录
- 用户权限
- 并发冲突处理
- 审批流

如果要放到组内服务器上，鉴权要由外层反向代理或内网环境负责。

### 4.4 还没有多进程安全写入

当前写回主要依赖本地文件系统与单进程服务假设。

这意味着：

- 单实例 `systemd + nginx` 反代是当前推荐姿态
- 不建议直接把 `review_server.py` 挂成多 worker 并发写同一批次目录
- 如果以后要上更强并发，需要补文件锁或服务端存储层

### 4.5 还没有真实 179 联机验证

当前仓库内的 179 相关能力，已经在本地完成：

- 单元测试验证
- 目录初始化 smoke
- `publish -> validate` smoke
- 本地 server 的批次发现、session、审核写回、bundle 导出 smoke

但还没有在真实 `179` 主机上确认：

- 网络路径
- 反向代理配置
- systemd 常驻
- 大批次真实数据的磁盘与权限表现

### 4.2 还没有中心化标签目录

时间轴上的“自定义类别”目前主要保存在浏览器 `localStorage`，不是服务端统一配置。

这意味着：

- 已保存的段落标记会跟批次一起保存，因为服务端写回了 `categoryId / categoryName / color`
- 但“右键菜单里有哪些可新建标签”目前仍然由当前浏览器本地设置决定

所以当前版本适合先统一批次格式和落盘位置，不适合宣称已经完成“多人统一标签治理”。

### 4.3 `segments/` 目录不是当前主存储

部分旧批次目录里仍然有：

```text
<batch_root>/segments/
```

这是早期遗留结构。当前页面实际使用的是：

```text
<batch_root>/review/timeline_annotations/
```

因此后续新批次不应再把 `segments/` 作为主契约。

## 5. 适合进入组内共享的边界

当前版本适合推进到：

- 组内共享一套服务
- 大家按统一批次格式接入自己的数据
- organizer / reviewer 在统一页面里以 reviewer 命名空间审阅
- 已审核结果与时间轴标注随批次目录持久化保存
- organizer 汇总同一轨迹上的多人审核差异
- operator 通过 `incoming -> validate -> publish` 第一版 CLI 流程接入外部批次

当前版本不适合直接承诺：

- 公网暴露
- 多团队正式平台
- 强一致多人协同
- 服务端统一标签治理

## 6. 推荐下一步

如果要把它放到组内服务器上，建议按下面顺序推进：

1. 先冻结批次目录合同、CSV 合同、输出位置
2. 再固定服务器上的 `batches-root`
3. 再决定是否加鉴权
4. 最后再推进统一标签目录与多人协作规则
