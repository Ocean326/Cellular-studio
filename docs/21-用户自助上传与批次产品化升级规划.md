# 用户自助上传与批次产品化升级规划

## 1. 任务定位

本次升级不是新做一套平台，而是把 `trajectory_annotation_studio` 已经存在的：

- `incoming -> validate -> publish`
- `published batch + source_result_root`
- `reviewer namespace + review/timeline 落盘`

从 operator 驱动的文件系统工作流，升级为面向用户的“自助上传 + 后端处理 + 可见性治理 + 删除治理 + 标注接入”产品能力。

本轮按照：

- `$delivery-conductor` 主线推进
- `$architect` 以 `spec-driven-architecture` 为主 lane
- `$bmad-orchestrator` 做阶段化交付拆解
- `$proposal-critique-refine` 做方案压测与收敛

当前结论：这是一个 **Level 3** 的增量升级项目。

## 2. 当前上下文与关键判断

### 2.1 已有能力

- 已有 Web 上传到 `incoming` 的能力，但只负责写 `payload.zip` 和 `upload_status.json`。
- 已有后端 `process_incoming_upload.py` / `server_batch_lib.py`，可以解包、校验标准 result 合同、发布 mounted batch。
- 已有批次发现能力，`review_server.py` 会通过 `result/` 或 `batch_meta.json.source_result_root` 发现 batch。
- 已有按 batch、按 reviewer 的 review / timeline / export 存储模型。

### 2.2 当前缺口

- 没有“原始 6 元组信令 / 原始 4 元组轨迹 -> 标准 studio result”的 adapter。
- 没有北京地区限制校验。
- 没有真正的平台用户身份，当前 `reviewer session` 不能承担 ownership / private visibility / delete 权限。
- 没有 batch 级 visibility 过滤与 path 级访问控制。
- 没有 upload / batch 的产品化删除语义。
- 没有异步任务、失败重试、处理状态面板。

### 2.3 核心判断

- 不能复用 `reviewer_id` 作为“上传拥有者”。
- 不能只在 `/api/batches` 里隐藏私有批次，`/batch-data/<batch>/...` 也必须做同一套访问控制。
- 不能让“上传物 = batch”，否则可见性切换、删除和回滚会很笨重。
- 删除第一版必须先做 soft delete / unpublish，不能直接物理删目录。

## 3. 需求范围

### 3.1 目标能力 A：6 元组信令上传

用户可以上传指定格式的 6 元组信令数据，后端完成：

- 格式校验
- 北京地区限制校验
- 标准化处理
- 生成 studio 可消费结果
- 按策略进入“仅自己可见”或“公开可见”
- 可选允许标注
- 可删除

### 3.2 目标能力 B：4 元组轨迹上传

用户可以上传指定格式的 4 元组轨迹数据，后端完成：

- 格式校验
- 标准化处理
- 生成 studio 可消费结果
- 按策略进入“仅自己可见”或“公开可见”
- 可选允许标注
- 可删除

## 4. 推荐目标架构

推荐升级为四层：

1. `ActorIdentity`
   - 平台真实用户身份
   - 用于 ownership、visibility、delete 权限
   - 来源建议为反向代理可信头，例如 `X-Forwarded-User`

2. `Upload / Processing`
   - 负责接收原始上传
   - 负责排队、校验、适配、生成标准资产

3. `Batch Catalog / Access Policy`
   - 负责哪些批次对谁可见
   - 负责 batch 生命周期与删除语义

4. `Studio Runtime`
   - 继续复用现有 `review_server.py + index.html + review_lib.py`
   - review / timeline / export 继续挂在 batch 上

### 4.2 生命周期

推荐拆成两段：

- 上传/处理生命周期：
  - `uploaded -> queued -> validating -> adapting -> asset_ready -> failed`
- 批次生命周期：
  - `active -> hidden/unpublished -> soft_deleted -> purged`

### 4.3 可见性策略

不要把“公开/私有/可标注”揉成一个枚举。

推荐两轴：

- `visibility_scope`
  - `owner_only`
  - `public`
- `annotation_mode`
  - `view_only`
  - `annotatable`

V1 建议只开放有限组合：

- `owner_only + annotatable`
- `public + view_only`

`public + annotatable` 放到后续版本，等身份和协作治理更稳后再开。

### 4.4 删除语义

推荐两阶段：

- `unpublish / hide`
  - 从批次列表消失
  - 阻止继续写 review
- `purge`
  - 延迟物理清理 asset / batch

V1 规则建议：

- 无引用、无 review 的上传可允许彻底删除
- 已发布且已有 review 的对象只允许 soft delete / archive

## 5. 存储布局建议

```text
shared/
  incoming/
    <upload_id>/
      upload_status.json
      payload.zip
      intake_report.json

  catalog/
    uploads/
      <upload_id>.json
    assets/
      <asset_id>.json
    batches/
      <batch_id>.json

  published/
    public/
      <batch_name>/
        batch_meta.json
        review/
        accepted_assets/
        review_exports/
    private/
      <owner_actor_id>/
        <batch_name>/
          batch_meta.json
          review/
          accepted_assets/
          review_exports/

datasets/
  user_assets/
    <owner_actor_id>/
      <asset_id>/
        v001/
          result/
```

补充说明：

- 目录扫描可以继续保留，但 `catalog` 应逐步成为主入口。
- `publish_batch()` 后续应同时写 `catalog/batches/<batch_id>.json`。
- 旧 batch 继续走 `discover_batches()` 兜底，避免打断已有数据。

## 8. 关键 Trade-off

### 8.1 同步还是异步

推荐：

- 上传创建同步
- 校验/适配/发布异步

原因：

- 解包、校验、分 UID、写 manifest/states 不应阻塞 HTTP 请求线程

### 8.2 上传物是否直接就是 batch

不推荐。

推荐：

- 上传物 -> 处理资产 -> 发布批次

原因：

- 同一资产后续可能切换可见性
- 删除/回滚/重发布更可控

### 8.3 删除是否直接物理删除

不推荐。

推荐：

- 先 hide
- 后 purge

原因：

- batch 下已经可能存在 review / timeline / export

## 10. 迭代路线

### 迭代一：上传域基础设施 + 4 元组 private 闭环

目标：

- 打通最小自助上传闭环

范围：

- `ActorIdentity`
- `UploadJob`
- `BatchCatalog`
- `trajectory4_adapter`
- 我的上传页
- 我的批次可见性过滤
- `owner_only + annotatable`

验收：

- 用户能上传合法 4 元组
- 后端能处理并生成 batch
- batch 只对本人可见
- 可以在 studio 打开
- 失败任务有明确错误

### 迭代二：6 元组信令 + 北京限制 + 标注接入

范围：

- `signal6_beijing_adapter`
- 北京区域限制
- 生成目标图层结果
- 允许进入现有 reviewer namespace 标注流

验收：

- 北京内合法样例通过
- 非北京样例拒绝且错误明确
- 打开 batch 后可以正常标注

### 迭代三：公开可见性 + 删除治理

范围：

- `public` 批次发布
- 公开批次列表
- soft delete / archive / hidden
- 删除权限控制

验收：

- private 不泄漏
- public 可共享浏览
- 删除后立即隐藏
- 已删除对象不可继续标注

### 迭代四：产品化补强

范围：

- 失败重试
- 任务详情
- 配额/限流
- 监控日志

- 回归测试补齐

## 12. 本轮结论

本次升级建议 **继续推进**，但推进方式必须是：

- 先补“用户上传与治理层”
- 再复用现有 mounted batch + review runtime
- 先做 `owner_only` 闭环
- 再开放 `public`

一句话概括：

> 不要重写 studio；应把它升级成“上传资产 -> 异步处理 -> 受控发布批次 -> 复用现有标注 runtime”的产品化架构。
