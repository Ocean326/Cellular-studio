# 179 服务器运行、权限与批次流转手册

## 1. 目的

这份文档不讨论产品愿景，只给出 179 服务器上的推荐运行和流转方式：

- 目录怎么摆
- 权限怎么切
- 批次怎么发布
- 用户怎么上传
- 结果怎么导出

## 2. 推荐目录

```text
/srv/trajectory_annotation_studio/
  app/
    releases/
    current
  datasets/
    cellular_quality/
  shared/
    published/
    incoming/
    exports/
  runtime/
    logs/
    tmp/
    uploads/
  workspaces/
```

## 3. 服务与系统身份

## 3.1 服务用户

- `ta_svc`
  - 运行 studio
  - 写 `review/`
  - 写 `accepted_assets/`
  - 写 `review_exports/`

## 3.2 管理组

- `ta_admin`
  - 部署代码
  - 发布批次
  - 做清理与恢复

## 3.3 上传组

- `ta_uploaders`
  - 允许上传到 `shared/incoming`

## 4. 权限原则

## 4.1 `app/`

- 所有者：`ta_svc:ta_admin`
- 权限：维护者可写，其他人只读或不可见
- 不允许普通成员直接改线上目录

## 4.2 `datasets/`

- 原则：尽量只读
- 推荐：
  - `ta_admin` 可写
  - `ta_svc` 可读
  - 其他用户不直接写

## 4.3 `shared/published/`

- 当前批次由 `ta_svc` 写入
- 用户通过 Web 使用，不通过 SSH 改这里

## 4.4 `datasets/`

- 大结果目录尽量只读
- `ta_svc` 只需要读权限
- 不要让服务进程改这里的 `raw.csv / snap.csv / od.csv / fmm.csv / line.csv`

## 4.5 `shared/incoming/`

- 每位上传者有自己的子目录
- 上传者只写自己的目录
- 推荐开启 `setgid`
- 不应被 `--batches-root` 直接扫描
- 只有在通过校验并发布后，才进入 `shared/published/`

## 4.6 `shared/exports/`

- 由服务进程生成
- 用户下载时只读

## 5. 推荐 systemd 服务

服务启动命令建议为：

```bash
python3 /srv/trajectory_annotation_studio/app/current/web/review_server.py \
  --host 127.0.0.1 \
  --port 8016 \
  --project-root /srv/trajectory_annotation_studio/app/current \
  --batches-root /srv/trajectory_annotation_studio/shared/published
```

说明：

- 进程只监听本机
- 外部访问通过 `nginx` 或 `caddy` 转发

## 6. 推荐反向代理职责

反向代理至少负责：

- HTTPS
- 访问控制
- 请求日志
- 上传体积限制
- 静态压缩

当前 `review_server.py` 不负责：

- 登录鉴权
- 文件上传
- 访问速率限制

另外当前推荐只跑：

- 一个 `review_server.py` 实例
- 一个可写共享目录

先不要把它部署成多 worker 同时写同一批次目录的模式。

## 7. 发布共享批次

推荐采用：

- 一个完整 cohort
- 多个 reviewer-facing shard

以 `signal100_10d_uid10000_round1` 为例：

## 7.1 准备结果数据

把大数据结果放到：

```text
/srv/trajectory_annotation_studio/datasets/cellular_quality/signal100_10d_uid10000_round1/result
```

## 7.2 创建发布批次目录

推荐不是只创建一个批次，而是创建 `10` 个 shard 批次，例如：

```text
/srv/trajectory_annotation_studio/shared/published/
  signal100_10d_uid10000_round1_b01of10/
  signal100_10d_uid10000_round1_b02of10/
  ...
  signal100_10d_uid10000_round1_b10of10/
```

每个 shard 对应 `1000 UID` 左右。

下面以 `b01of10` 为例。

```text
/srv/trajectory_annotation_studio/shared/published/signal100_10d_uid10000_round1_b01of10/
  batch_meta.json
  review/
  accepted_assets/
  review_exports/
```

## 7.3 写入 `batch_meta.json`

至少包含：

```json
{
  "name": "signal100_10d_uid10000_round1_b01of10",
  "label": "signal100_10d_uid10000_round1_b01of10",
  "version": "v1",
  "created_at": "2026-04-20T08:00:00Z",
  "keywords": ["signal100", "10d", "uid10000", "shared", "b01of10"],
  "cohort_id": "signal100_10d_uid10000_round1",
  "shard_index": 1,
  "shard_count": 10,
  "source_result_root": "/srv/trajectory_annotation_studio/datasets/cellular_quality/signal100_10d_uid10000_round1/result",
  "result_mode": "mounted",
  "uid_count": 1000
}
```

## 7.4 打开服务后验证

检查：

- `/api/batches` 中出现该批次
- `/api/health?batch=signal100_10d_uid10000_round1_b01of10` 正常
- `manifest.json` 能被前端加载

## 8. 用户上传自己的数据

## 8.1 推荐上传入口

```text
/srv/trajectory_annotation_studio/shared/incoming/<user>/<upload_name>/
```

说明：

- 这个目录只用于上传和 staging
- 不是 studio 直接扫描的 `published` 批次根

## 8.2 推荐上传的两种类型

### A. 标准批次上传

目录已经接近：

```text
<upload_name>/
  batch_meta.json
  result/
    manifest.json
    <uid>/
```

### B. 原始数据上传

目录可能是：

```text
<upload_name>/
  raw/
  maps/
  notes.md
```

这种不直接发布，需要 adapter 转换。

## 8.3 建议的发布前校验清单

- `batch_meta.json` 是否存在
- `result/manifest.json` 是否存在
- `uids` 是否非空
- `result/<uid>/` 是否真实存在
- CSV 是否可读
- 是否含非法超大缓存文件
- 是否含路径穿越或软链越权

## 8.4 建议的发布动作

### 对标准批次

- 直接复制或挂载到 `datasets/`
- 在 `published/` 建轻量批次入口

### 对原始数据

- 先在 GitHub 仓里提供 adapter
- 在 `workspaces/` 或离线环境转换
- 转成标准批次后再发布

## 9. 导出流程

## 9.1 当前已可直接使用的导出

`POST /api/export/accepted`

结果位置：

```text
<batch_root>/accepted_assets/reviewers/<reviewer_id>/
```

适合：

- 高质量 `accept` 子集导出

## 9.2 建议新增的导出

建议增加 `reviewer bundle export`：

- 导出某个 reviewer 的全部已标注 UID
- 附带原始数据与中间结果

推荐输出到：

```text
/srv/trajectory_annotation_studio/shared/exports/reviewers/<reviewer_id>/<bundle_name>.zip
```

## 9.3 建议的导出目录

解压后建议结构：

```text
<bundle_name>/
  bundle_manifest.json
  reviews/
  timeline_annotations/
  samples/
    <uid>/
      raw.csv
      snap.csv
      od.csv
      fmm.csv
      line.csv
      review.json
      timeline_annotations.json
```

## 10. GitHub 更新到 179 的发布流程

推荐：

1. GitHub 合并到 `main`
2. 在 179 上拉取或同步到新 release 目录
3. 切换 `app/current`
4. 重启 `systemd`
5. 用 `/api/health` 和 `/api/batches` 冒烟验证

不推荐：

- 直接在运行中的 `current` 目录里手改文件

## 11. 备份建议

至少分 3 类：

### 代码备份

- GitHub 即主备份

### 数据备份

- `datasets/` 做周期性备份

### 标注结果备份

- `shared/published/*/review`
- `shared/published/*/accepted_assets`
- `shared/published/*/review_exports`

其中标注结果建议比大数据更高频备份。

## 12. 本轮最先需要补的脚本

如果要把这套流程真正跑起来，建议优先补 4 个脚本：

1. `scripts/validate_published_batch.py`
   - 校验批次是否满足共享挂载要求

2. `scripts/publish_server_batch.py`
   - 把 `incoming` 或 `datasets` 中的结果发布为 `published batch`

3. `scripts/export_reviewer_bundle.py`
   - 打包某个 reviewer 的全部已标注数据与原始数据

4. `scripts/list_server_batches.py`
   - 列出 `published`、`incoming`、`archived` 批次摘要

## 13. 运行手册结论

这套 179 方案真正要守住的，不是某一个脚本，而是 3 条规则：

1. 大数据只读挂载，review 写回单独存放。
2. 上传先进 `incoming`，不要直写 `published`。
3. 代码更新走 GitHub -> release -> current，不直接改线上运行目录。
