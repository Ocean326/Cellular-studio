# 179 详细操作手册

## 1. 文档目标

这份文档面向 `operator`，覆盖三件事：

1. 发布首轮真实批次
2. 处理用户上传的 zip
3. 导出 reviewer 标注结果

默认根目录：

```text
/srv/trajectory_annotation_studio
```

如果当前账号没有 `sudo`，本手册中的根目录可以整体替换为：

```text
/home/<user>/trajectory_annotation_studio_pilot
```

本轮真实试运行实际使用的就是这条 `user-space pilot` 路径。

## 2. 首轮真实批次发布

## 2.1 准备数据根

推荐把真实数据放在：

```text
/srv/trajectory_annotation_studio/datasets/cellular_quality/
```

在 `user-space pilot` 下，对应为：

```text
/home/<user>/trajectory_annotation_studio_pilot/datasets/cellular_quality/
```

如果本轮 cohort 名为：

```text
signal100_10d_uid10000_round1
```

推荐目录为：

```text
/srv/trajectory_annotation_studio/datasets/cellular_quality/signal100_10d_uid10000_round1/
  shards/
    b01of10/result/
    b02of10/result/
    ...
    b10of10/result/
```

重要说明：

- 当前 `publish_server_batch.py` 不会自动把一个全量 `result/` 切成 `10` 个 shard
- 也就是说，`10 x 1000 UID` 的前提是你已经准备好了每个 shard 的独立 `result root`
- 不要把同一个全量 `result/` 重复挂成 10 个批次，否则前端会看到相同数据

## 2.2 先验证一个 shard

先验证 `b01of10`：

```bash
python3 /srv/trajectory_annotation_studio/app/current/scripts/publish_server_batch.py \
  --published-root /srv/trajectory_annotation_studio/shared/published \
  --batch-name signal100_10d_uid10000_round1_b01of10 \
  --source-result-root /srv/trajectory_annotation_studio/datasets/cellular_quality/signal100_10d_uid10000_round1/shards/b01of10/result \
  --label "signal100 round1 b01of10" \
  --keywords signal100,10d,uid10000,round1,b01of10 \
  --cohort-id signal100_10d_uid10000_round1 \
  --shard-index 1 \
  --shard-count 10
```

然后检查：

```bash
python3 /srv/trajectory_annotation_studio/app/current/scripts/validate_published_batch.py \
  --batch-root /srv/trajectory_annotation_studio/shared/published/signal100_10d_uid10000_round1_b01of10 \
  --require-review-reference
```

如果没问题，再访问：

```bash
curl http://127.0.0.1:8016/api/batches
curl "http://127.0.0.1:8016/api/health?batch=signal100_10d_uid10000_round1_b01of10"
```

如果当前是 `ssh -L` 隧道模式，则在本机执行：

```bash
curl http://127.0.0.1:18016/api/batches
curl "http://127.0.0.1:18016/api/health?batch=signal100_10d_uid10000_round1_b01of10"
```

## 2.3 批量发布 10 个 shard

当单 shard 成功后，再批量发布：

```bash
for i in 01 02 03 04 05 06 07 08 09 10; do
  python3 /srv/trajectory_annotation_studio/app/current/scripts/publish_server_batch.py \
    --published-root /srv/trajectory_annotation_studio/shared/published \
    --batch-name signal100_10d_uid10000_round1_b${i}of10 \
    --source-result-root /srv/trajectory_annotation_studio/datasets/cellular_quality/signal100_10d_uid10000_round1/shards/b${i}of10/result \
    --label "signal100 round1 b${i}of10" \
    --keywords signal100,10d,uid10000,round1,b${i}of10 \
    --cohort-id signal100_10d_uid10000_round1 \
    --shard-index ${i#0} \
    --shard-count 10
done
```

发布后：

```bash
curl http://127.0.0.1:8016/api/batches
```

## 3. 处理用户上传的 zip

## 3.1 Web 上传后查看 incoming

服务运行后，可以用：

```bash
curl http://127.0.0.1:8016/api/admin/incoming
```

如果当前走 `ssh -L 18016:127.0.0.1:18016 ...`，则本机侧改为：

```bash
curl http://127.0.0.1:18016/api/admin/incoming
```

如果要直接从命令行模拟接入者上传，可用：

```bash
curl -X POST "http://127.0.0.1:8016/api/admin/incoming/upload?name=min_batch.zip" \
  -H 'Content-Type: application/zip' \
  -H 'X-Upload-Filename: min_batch.zip' \
  --data-binary @/path/to/min_batch.zip
```

每个上传条目会落在：

```text
/srv/trajectory_annotation_studio/shared/incoming/<upload_id>/
  payload.zip
  upload_status.json
```

## 3.2 对某个上传条目做 intake

假设当前 upload id 为：

```text
20260415T021500Z_abc12345
```

执行：

```bash
python3 /srv/trajectory_annotation_studio/app/current/scripts/process_incoming_upload.py \
  --upload-root /srv/trajectory_annotation_studio/shared/incoming/20260415T021500Z_abc12345 \
  --workspace-root /srv/trajectory_annotation_studio/workspaces/incoming \
  --published-root /srv/trajectory_annotation_studio/shared/published \
  --batch-name incoming_demo_b01 \
  --label "incoming demo b01" \
  --keywords incoming,demo \
  --require-review-reference \
  --clean-workspace \
  --publish
```

它会做：

1. 解包 `payload.zip`
2. 自动寻找 `result/manifest.json`
3. 做结构校验
4. 如果 `--publish`，直接发布到 `shared/published`
5. 写回：

```text
<upload_root>/intake_report.json
<upload_root>/upload_status.json
```

## 3.3 只校验不发布

如果先只想验证：

```bash
python3 /srv/trajectory_annotation_studio/app/current/scripts/process_incoming_upload.py \
  --upload-root /srv/trajectory_annotation_studio/shared/incoming/20260415T021500Z_abc12345 \
  --workspace-root /srv/trajectory_annotation_studio/workspaces/incoming \
  --require-review-reference \
  --clean-workspace
```

## 4. 导出 reviewer 标注结果

## 4.0 先做一条最小 review

如果只是验证整条链路，推荐最小顺序：

1. 建 reviewer session
2. 写一条 review
3. 可选写一条最小 timeline annotation
4. 再导出 reviewer bundle

注意：

- 第 2 步和第 4 步不要并行触发
- `review` 还没落盘就立刻 `export`，可能得到一个空 bundle
- 正确姿态是严格串行：`session -> review -> annotation -> export`

例如：

```bash
curl -X POST http://127.0.0.1:8016/api/reviewers/session \
  -H 'Content-Type: application/json' \
  -d '{
    "batch": "incoming_demo_b01",
    "display_name": "Alice Demo"
  }'

curl -X POST http://127.0.0.1:8016/api/reviews \
  -H 'Content-Type: application/json' \
  -d '{
    "batch": "incoming_demo_b01",
    "uid": "1001",
    "decision": "accept",
    "reviewer_id": "alice-demo",
    "reviewer_name": "Alice Demo",
    "reference_source": "line.csv",
    "notes": "minimal api review"
  }'

curl -X POST http://127.0.0.1:8016/api/timeline-annotations \
  -H 'Content-Type: application/json' \
  -d '{
    "batch": "incoming_demo_b01",
    "uid": "1001",
    "reviewer_id": "alice-demo",
    "reviewer_name": "Alice Demo",
    "pins": [{"id": "p1", "time": 1, "layerKey": "raw", "label": "demo pin"}],
    "segments": [{"id": "s1", "categoryId": "demo", "categoryName": "Demo Segment", "color": "#ff6600", "startTime": 0, "endTime": 1}]
  }'
```

## 4.1 通过 API 导出 reviewer bundle

先确保 reviewer 已在某个批次写入审核。

如果你刚刚才写完 review，先确认 `/api/reviews` 已经返回成功，再执行导出。

然后可调用：

```bash
curl -X POST http://127.0.0.1:8016/api/export/reviewer-bundle \
  -H 'Content-Type: application/json' \
  -d '{
    "batch": "signal100_10d_uid10000_round1_b01of10",
    "reviewer_id": "alice",
    "clean": true,
    "bundle_name": "alice_round1_b01_bundle",
    "create_zip": true
  }'
```

导出内容包含：

- reviewer 的最新审核结果
- reviewer 的 timeline annotations
- 对应样本的 `raw.csv / snap.csv / od.csv / fmm.csv / line.csv`

## 4.2 导出后检查

检查返回 JSON 里的：

- `bundle_root`
- `zip_path`

确保：

```bash
ls -lah <bundle_root>
ls -lah <zip_path>
```

## 6. 失败时先看哪里

### 6.1 `/api/batches` 没有新批次

优先检查：

- `batch_meta.json` 是否存在
- `source_result_root` 是否存在
- `manifest.json` 是否可读

### 6.2 `process_incoming_upload.py` 失败

优先检查：

- zip 内是否真的有 `result/manifest.json`
- zip 是否包含非法路径
- `line.csv` 或 `fmm.csv` 是否缺失

### 6.3 前端能开但没有轨迹

优先检查：

- 对应 `result/<uid>/` 是否真的存在
- `manifest.json.uids` 是否非空
- `source_result_root` 是否指到错目录

### 6.4 179 上没有 sudo，文档里的 `/srv` 走不通

这是首轮常见问题，不是代码 bug。

处理方式：

- 不要强行卡在 `/srv + systemd + nginx`
- 先切换到 `/home/<user>/trajectory_annotation_studio_pilot`
- 用高位端口起用户态服务
- 本机通过 `ssh -L` 隧道验证整个流程

## 7. 本地最小 smoke（已实跑）

下面这组命令已经在本地临时目录实跑通过，覆盖：

- 初始化 `179` 目录布局
- 发布一个最小 shard
- 校验一个已发布 batch
- 处理一个 incoming zip 并发布
- 启动 `review_server.py` 后检查 `health / batches / incoming`

直接执行：

```bash
set -euo pipefail

REPO=/Users/ocean/Documents/Playground/Cellular-projects/trajectory_annotation_studio
TMP_ROOT=$(mktemp -d /tmp/tas179_smoke.XXXXXX)
SERVER_ROOT="$TMP_ROOT/srv/trajectory_annotation_studio"
RELEASE_NAME=smoke_r1
RELEASE_ROOT="$SERVER_ROOT/app/releases/$RELEASE_NAME"
PORT=18116

python3 "$REPO/scripts/init_179_server_layout.py" \
  --root "$SERVER_ROOT" \
  --release-name "$RELEASE_NAME" \
  --create-current-link

rsync -a --exclude '.git' --exclude '__pycache__' --exclude '.pytest_cache' \
  "$REPO/" "$RELEASE_ROOT/"

DATASET_RESULT_ROOT="$SERVER_ROOT/datasets/cellular_quality/smoke_round1/shards/b01of01/result"
mkdir -p "$DATASET_RESULT_ROOT/1001" "$DATASET_RESULT_ROOT/1002"

cat > "$DATASET_RESULT_ROOT/manifest.json" <<'JSON'
{"uids":["1001","1002"],"layers":["raw","snap","od","fmm","line"],"states":{"1001":["road"],"1002":["stay"]}}
JSON

cat > "$DATASET_RESULT_ROOT/states_index.json" <<'JSON'
{"1001":["road"],"1002":["stay"]}
JSON

for uid in 1001 1002; do
  cat > "$DATASET_RESULT_ROOT/$uid/raw.csv" <<'CSV'
a,b
1,2
CSV
  cat > "$DATASET_RESULT_ROOT/$uid/snap.csv" <<'CSV'
a,b
1,2
CSV
  cat > "$DATASET_RESULT_ROOT/$uid/od.csv" <<'CSV'
a,b
1,2
CSV
done

cat > "$DATASET_RESULT_ROOT/1001/line.csv" <<'CSV'
a,b
1,2
CSV

cat > "$DATASET_RESULT_ROOT/1002/fmm.csv" <<'CSV'
a,b
1,2
CSV

python3 "$RELEASE_ROOT/scripts/publish_server_batch.py" \
  --published-root "$SERVER_ROOT/shared/published" \
  --batch-name smoke_round1_b01of01 \
  --source-result-root "$DATASET_RESULT_ROOT" \
  --label "smoke round1 b01of01" \
  --keywords smoke,round1 \
  --cohort-id smoke_round1 \
  --shard-index 1 \
  --shard-count 1

python3 "$RELEASE_ROOT/scripts/validate_published_batch.py" \
  --batch-root "$SERVER_ROOT/shared/published/smoke_round1_b01of01" \
  --require-review-reference

UPLOAD_ROOT="$SERVER_ROOT/shared/incoming/20260415T021500Z_smoke1234"
mkdir -p "$UPLOAD_ROOT"

cat > "$UPLOAD_ROOT/upload_status.json" <<'JSON'
{"upload_id":"20260415T021500Z_smoke1234","status":"uploaded","original_name":"incoming_smoke_bundle.zip"}
JSON

python3 - <<'PY' "$UPLOAD_ROOT/payload.zip"
import sys, zipfile
zip_path = sys.argv[1]
with zipfile.ZipFile(zip_path, "w") as zf:
    zf.writestr("result/manifest.json", '{"uids":["2001","2002"]}\n')
    zf.writestr("result/states_index.json", '{"2001":["road"],"2002":["stay"]}\n')
    zf.writestr("result/2001/raw.csv", "a,b\n1,2\n")
    zf.writestr("result/2001/line.csv", "a,b\n1,2\n")
    zf.writestr("result/2002/raw.csv", "a,b\n1,2\n")
    zf.writestr("result/2002/fmm.csv", "a,b\n1,2\n")
PY

python3 "$RELEASE_ROOT/scripts/process_incoming_upload.py" \
  --upload-root "$UPLOAD_ROOT" \
  --workspace-root "$SERVER_ROOT/workspaces/incoming" \
  --published-root "$SERVER_ROOT/shared/published" \
  --batch-name incoming_smoke_b01 \
  --label "incoming smoke b01" \
  --keywords incoming,smoke \
  --require-review-reference \
  --clean-workspace \
  --publish

python3 "$RELEASE_ROOT/web/review_server.py" \
  --host 127.0.0.1 \
  --port "$PORT" \
  --project-root "$RELEASE_ROOT" \
  --batches-root "$SERVER_ROOT/shared/published" \
  --incoming-root "$SERVER_ROOT/shared/incoming" \
  --upload-max-bytes 10485760
```
