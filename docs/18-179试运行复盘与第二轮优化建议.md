# 179 试运行复盘与第二轮优化建议

## 1. 这轮已经真正证明了什么

首轮真实执行之后，可以把当前结论压缩成 4 句话：

1. `Cellular-studio` 已经可以在没有 `sudo` 的 `179` 账号下，以 `user-space pilot` 方式稳定跑起来。
2. 一个真实 `1000 UID * 10days` shard 已经完成 `publish -> health -> review -> export` 闭环。
3. agent / 外部方法接入方已经可以按一份最小结果 zip，通过 `incoming -> validate -> publish` 被挂到 studio。
4. 当前主要问题已经不是“功能有没有”，而是“流程有没有被 operator 足够稳定地重复执行”。

这意味着第二轮的重心不该是盲目加新功能，而该是：

- 把真实通过的路径收敛成更稳的操作面
- 把容易误用的步骤显式化
- 把 `pilot` 和“正式开放”之间的边界说清楚

## 2. 当前流程里最不合适的地方

### 2.1 文档默认姿态仍偏“正式服务器”

虽然 `14/15/16` 已经补了 `user-space pilot` 路线，但读者第一眼仍容易把 `/srv + systemd + nginx` 误以为是本轮默认路径。

不合适点：

- 它会让 operator 在没有 `sudo` 的前提下多绕一圈
- 它也会让接入者误以为已经有公网式访问体验

建议：

- 后续继续把 `pilot 默认路径` 写得更前置
- 只有在确认拿到 `sudo` 和 nginx 入口时，才切换“正式部署”口径

### 2.2 operator 成功依赖“记得按顺序做”

本轮已经证明：

- `publish -> validate`
- `review -> export`

都必须串行。

不合适点：

- 这个约束如果只写在说明文字里，下一次很容易再次被误操作

建议：

- 第二轮把这两条改成 checklist
- 最好再补一个轻量 wrapper 或 shell recipe，减少现场自由发挥

### 2.3 shard 准备责任仍然太靠人脑记忆

当前 `publish_server_batch.py` 不负责自动切 shard，这是合理的。

但风险在于：

- operator 容易误以为“有一个全量 result root 就能直接发布 10 个 shard”

建议：

- 第二轮固定 `dataset cohort -> shards/ -> published` 的目录说明
- 在手册里把“先准备独立 shard result root”写成强前置条件

### 2.4 访问方式对 pilot 足够，但对多人协作还不够友好

当前方式是：

- 服务监听 `127.0.0.1`
- 使用者通过 `ssh -L` 隧道访问

这对首轮是对的，因为：

- 安全边界简单
- 不依赖额外运维能力
- 能先把真实工作流跑通

但它不适合作为多人长期协作默认姿态，因为：

- 每个人都要自己配隧道
- 连接状态不够可观测
- 文档和实际体验之间仍有运维门槛

## 3. 第二轮建议的优先级

### P0：立刻做，提升复用稳定性

1. 固化 `user-space pilot` operator 启动方式
   - 目标：减少手敲 `nohup`、忘记记录 pid、忘记看 log 的风险
2. 把串行步骤改成显式 checklist
   - 目标：把“经验”变成“流程”
3. 再做一轮双 reviewer / 双 agent 共享试运行
   - 目标：验证 reviewer namespace 的多人体验

### P1：下一轮再做，提升开放能力

1. 补 `/srv + systemd + nginx` 正式化落地
2. 规划更友好的访问入口
3. 为更大批次补 I/O 与并发观察

### P2：先不要急着做

1. 不要现在就做复杂权限系统
2. 不要现在就做上传后自动上架
3. 不要现在就做复杂审批流
4. 不要在 pilot 尚未稳固前，把流程拆成太多角色

这些都不是没价值，而是它们会让当前最需要打稳的主链路变得分散。

## 4. 第二轮推荐的默认操作面

如果下一轮还要继续以 `179` 做真实协作，建议默认这样执行：

1. operator 只维护一个 `pilot root`
   - `/home/<user>/trajectory_annotation_studio_pilot`
2. 每次 release 只切 `app/current`
3. 每次发布先做单 shard 验证，再放大
4. 每次接入先走 `incoming -> validate -> publish`
5. 每次导出都要求 review 已落盘，再执行 export
6. 每轮结束都追加一份试运行记录

这样做的好处是：

- 不会把 “代码发布”“数据发布”“上传接入”“标注导出” 混成一团
- 也更方便后面把某一段迁到正式服务姿态

## 5. 我们现在可以怎么对外描述这套系统

当前最准确的说法是：

`Cellular-studio` 已具备一套可在受控服务器上运行的共享标注 pilot 流程，支持真实批次发布、接入者上传、reviewer 命名空间标注和 reviewer bundle 导出；首轮已在 `179` 上完成真实 1000 UID shard 试运行，但当前仍属于 operator-controlled pilot，不应包装为“正式开放平台”。

这个说法的好处是：

- 不低估当前成果
- 也不夸大当前开放程度

## 6. 建议的下一次验收口径

第二轮如果要算“明显更稳”，建议至少满足这 4 条：

1. 新 operator 只看手册，也能复现一次 `user-space pilot` 启动
2. 两个 reviewer 或两个 agent 能同时在不同 batch 上完成最小写入
3. 一个真实 incoming zip 能再次成功被发布
4. 复盘文档中不再出现“靠记忆操作才成功”的步骤
