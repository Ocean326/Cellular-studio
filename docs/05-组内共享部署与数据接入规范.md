# trajectory_annotation_studio 组内共享部署与数据接入规范

## 1. 目的

这份文档只回答两件事：

1. 组内共享时，服务器上应该怎么摆目录
2. 别人自己的数据要按什么格式接入，输出又会写到哪里

这里写的是当前实现真实支持的格式，不是理想中的未来接口。

## 2. 推荐部署方式

### 2.1 运行模式

组内共享时，统一使用：

- 一份代码仓
- 一个共享的 `batches-root`
- 每个数据集/方法批次占一个子目录

不推荐每个人各自改 `--result-root` 临时起服务，那样后续很难统一审阅和回收结果。

### 2.2 推荐服务器目录

推荐把“代码”和“数据批次”分离：

```text
/srv/trajectory_annotation_studio/
  app/
    trajectory_annotation_studio/   # 代码仓
  shared/
    batches/                        # 统一批次根
      <batch_name>/
        batch_meta.json
        result/                     # 或者通过 batch_meta.json 指向外部 source_result_root
        review/
        accepted_assets/
        review_exports/
        track_manifest.json         # 推荐
        label_catalog.json          # 推荐
        source_batch.json           # 可选
```

其中：

- `app/trajectory_annotation_studio/` 是代码
- `shared/batches/` 是真正共享的数据与写回目录
- 每个 `<batch_name>` 都是一个可独立打开的批次

### 2.3 启动命令

推荐命令：

```bash
cd /srv/trajectory_annotation_studio/app/trajectory_annotation_studio
python3 web/review_server.py \
  --host 0.0.0.0 \
  --port 8016 \
  --project-root /srv/trajectory_annotation_studio/app/trajectory_annotation_studio \
  --batches-root /srv/trajectory_annotation_studio/shared/batches
```

如果只是本机验证，也可以继续使用本地路径。

## 3. 批次根目录合同

### 3.1 组内共享时建议采用的最小标准结构

当前服务实际可以兼容比这更松的目录，但为了组内共享、统一运维和后续回收结果，建议把最小标准结构固定为：

```text
<batch_root>/
  batch_meta.json
  review/
    system/
      reviewer_registry.json
    reviewers/
      <reviewer_id>/
        profile.json
        reviews/
          ledger.jsonl
          latest_reviews.json
        timeline_annotations/
          ledger.jsonl
          <uid>.json
    aggregate/
      stats.json
      by_uid/
        <uid>.json
  accepted_assets/
    reviewers/
      <reviewer_id>/
  review_exports/
    aggregate/
  result/
    manifest.json                  # 强烈建议提供
    <uid_1>/
      <layer_a>.csv
      <layer_b>.csv
    <uid_2>/
      ...
```

说明：

- `batch_meta.json` 推荐作为标准入口
- `review/` 是写回根目录
- `review/reviewers/` 是当前 reviewer 命名空间主存储
- `review/aggregate/` 用于多人 latest 汇总与冲突摘要
- `result/` 是前端实际读轨迹数据的地方
- `result/<uid>/` 下不要求固定五件套，但至少要有 manifest 声明的可展示图层之一
- `accepted_assets/reviewers/<reviewer_id>/` 是按 reviewer 导出根
- `review_exports/aggregate/` 是 organizer 侧汇总导出根

### 3.2 支持外部结果目录挂载

如果不想把 `result/` 复制到批次里，可以把 `result/` 省略，然后在 `batch_meta.json` 里写：

- `source_result_root`

当前 `review_server.py` 会优先尝试：

1. `<batch_root>/result`
2. `batch_meta.json.source_result_root`

也就是说，下面这种结构是被当前实现支持的：

```text
<batch_root>/
  batch_meta.json
  review/
  accepted_assets/                 # 可先不存在，首次 reviewer 导出时生成
  review_exports/                  # 可先不存在，首次 aggregate 导出时生成
```

同时 `batch_meta.json` 中写：

```json
{
  "name": "20260413_v1_teamA_demo",
  "label": "Team A Demo",
  "source_result_root": "/data/teamA/demo_result"
}
```

### 3.3 当前推荐哪些伴随文件

推荐但非强制：

- `track_manifest.json`
- `label_catalog.json`
- `source_batch.json`

当前前端真正强依赖的只有：

- `result/manifest.json` 或 `result/` 目录扫描结果

## 4. `batch_meta.json` 规范

### 4.1 必填与推荐字段

当前推荐固定以下字段：

```json
{
  "name": "20260413_v1_teamA_demo",
  "label": "20260413_v1_teamA_demo",
  "version": "v1",
  "created_at": "2026-04-13T08:00:00Z",
  "keywords": ["annotation-studio", "teamA", "demo"],
  "source_result_root": "/data/teamA/demo_result",
  "result_mode": "mounted",
  "uid_count": 1200
}
```

字段说明：

- `name`
批次唯一名，建议与目录名一致
- `label`
前端展示名
- `version`
批次版本
- `created_at`
批次生成时间，建议 UTC ISO8601
- `keywords`
便于后续检索与管理
- `source_result_root`
当 `result/` 不在批次内时使用
- `result_mode`
推荐写明 `copied` / `symlink` / `mounted`
- `uid_count`
推荐补充，用于前端摘要显示

### 4.2 命名建议

批次命名建议固定为：

```text
YYYYMMDDTHHMMSSZ_v{version}_{keywords}
```

或者在多人共享场景下，用更容易阅读的本地时间版本：

```text
YYYYMMDD_v{version}_{team_or_owner}_{keywords}
```

只要满足：

- 唯一
- 稳定
- 可追溯

即可。

## 5. `result/manifest.json` 规范

### 5.1 最小推荐格式

当前最推荐提供的 `manifest.json` 结构：

```json
{
  "ui_mode": "chain2",
  "uids": ["1001", "1002"],
  "layers": ["raw", "snap", "od", "fmm", "line"],
  "states": {
    "1001": ["road", "stay"],
    "1002": ["stay"]
  },
  "filter_state_options": ["road", "subway", "railway", "unmatch", "stay"]
}
```

说明：

- `uids`
前端 UID 列表主来源
- `layers`
当前批次建议展示的图层顺序
- `states`
每个 UID 的预计算状态标签，能显著加快筛选
- `filter_state_options`
左侧筛选区显示哪些标签

### 5.1.1 异构图层推荐扩展

如果批次不是旧版 `raw/snap/od/fmm/line`，推荐显式提供：

- `layer_specs`
- `review_reference_files`
- `hide_review_panel`
- `time_scrubber_preferred_layers`

示例见：

- [manifest.trajectory_layers.example.json](/Users/ocean/Documents/Playground/Cellular-projects/trajectory_annotation_studio/docs/examples/manifest.trajectory_layers.example.json)

其中：

- `layer_specs.<layer>.filename`
  指定该图层真实读取哪个 CSV
- `layer_specs.<layer>.kind`
  指定该图层按 `gps / signal / default / od / line / stations` 中哪一种语义渲染；其中 `od` 会按 OD 点层处理，默认不要求提供 polyline 连线
- `review_reference_files`
  指定 review `accept` 时允许引用哪些文件；未提供时回退到 `line.csv / fmm.csv`

### 5.2 可选 UI 字段

当前前端还能识别这些可选字段：

- `ui_mode`
`chain2` 或 `sim_signal`
- `title`
- `filter_source_label`
- `layer_styles`
- `layer_visibility`
- `layer_labels`
- `point_status_types`
- `point_status_styles`
- `long_jump_threshold_m`

如果不提供，前端会退回默认配置。

## 6. 支持的数据模式

### 6.1 模式 A：`chain2`

适用于当前 `cellular_quality` 风格结果，推荐图层：

- `raw`
- `snap`
- `od`
- `fmm`
- `line`

### 6.1.1 `raw.csv` / `snap.csv`

最小推荐字段：

- `latitude`
- `longitude`

强烈建议补充：

- `uid`
- `cid` 或 `CID`
- `t_in`
- `t_out`

当前页面会用这些字段做：

- 地图点和线显示
- 时间标签
- 停留时长
- 基站号显示

### 6.1.2 `od.csv`

最小推荐字段：

- `start_latitude`
- `start_longitude`

强烈建议补充：

- `end_latitude`
- `end_longitude`
- `start_time`
- `end_time`
- `is_stationary`

### 6.1.3 `fmm.csv`

最小推荐字段：

- `latitude`
- `longitude`

强烈建议补充：

- `t_in`
- `segment_start_time`
- `segment_end_time`
- `match_type`
- `segment_idx`
- `point_order`
- `is_stationary`

### 6.1.4 `line.csv`

最小推荐字段：

- `latitude`
- `longitude`

强烈建议补充：

- `match_type`
- `segment_idx`
- `point_order`
- `segment_start_time`
- `segment_end_time`
- `is_stationary`

### 6.1.5 `accept` 的硬性要求

如果要支持整轨审核里的 `accept`，当前后端要求：

- `line.csv` 或 `fmm.csv` 至少有一个存在
- 且该文件非空

否则 `accept` 会直接报错。

### 6.2 模式 B：`sim_signal`

适用于 `GPS / 模拟信令 / 基站` 数据，推荐图层：

- `gps`
- `signal`
- `stations`

### 6.2.1 `gps.csv`

最小推荐字段：

- `latitude`
- `longitude`

强烈建议补充：

- `timestamp_ms` 或 `timestamp` 或 `time`
- `status`

### 6.2.2 `signal.csv`

最小推荐字段：

- `latitude`
- `longitude`

强烈建议补充：

- `CID`
- `procedureStart`
- `procedureEnd`

### 6.2.3 `stations.csv`

最小推荐字段：

- `latitude`
- `longitude`

强烈建议补充：

- `CID`

## 7. 输出与写回位置规范

### 7.1 整轨审核输出

固定写到：

```text
<batch_root>/review/ledger.jsonl
<batch_root>/review/latest_reviews.json
```

记录结构固定为：

```json
{
  "uid": "1001",
  "sample_id": "1001",
  "decision": "accept",
  "reviewer": "alice",
  "timestamp": "2026-04-13T08:10:00Z",
  "notes": "整体可信",
  "reference_source": "line.csv"
}
```

### 7.2 时间轴标记输出

固定写到：

```text
<batch_root>/review/timeline_annotations/<uid>.json
<batch_root>/review/timeline_annotations/ledger.jsonl
```

当前单 UID 文件结构：

```json
{
  "uid": "1001",
  "sample_id": "1001",
  "updated_at": "2026-04-13T08:20:00Z",
  "pins": [
    {
      "id": "1677651200-12",
      "time": 1677651200,
      "layerKey": "raw",
      "label": "2023-03-01 08:00:00"
    }
  ],
  "segments": [
    {
      "id": "stay:1677651200:1677654800",
      "categoryId": "stay",
      "categoryName": "停留",
      "color": "#7c3aed",
      "startTime": 1677651200,
      "endTime": 1677654800
    }
  ]
}
```

### 7.3 accepted 导出

固定写到：

```text
<batch_root>/accepted_assets/
```

导出内容固定为：

```text
accepted_assets/
  samples/
    <sample_id>/
      raw.csv
      line.csv | fmm.csv
      review.json
  accepted_reviews.jsonl
  export_manifest.json
```

## 8. 推荐接入流程

对于组内其他人的自定义数据，推荐固定流程：

1. 先把自己的结果整理成 `result/<uid>/*.csv`
2. 生成 `manifest.json`
3. 新建一个 `<batch_root>`
4. 写好 `batch_meta.json`
5. 初始化 `review/ledger.jsonl` 和 `review/latest_reviews.json`
6. 把这个批次放到共享 `batches-root` 下
7. 刷新前端并切换到该批次

如果第 5 步暂时没做，当前实现通常也能在第一次写 review 时自动补齐；但为了共享环境稳定，仍建议预先初始化。

## 9. 不建议的做法

下面这些做法当前不建议：

- 把真实写回结果保存到浏览器本地而不是批次目录
- 把共享批次直接塞在代码仓里混合管理
- 继续把 `segments/` 当成正式输出位置
- 不给 `manifest.json` 就上大批量数据
- 不提供 `line.csv/fmm.csv` 却要求 reviewer 可以 `accept`
