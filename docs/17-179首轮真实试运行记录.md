## 5. 本轮暴露出的流程问题

### 5.1 文档原先默认假设有 sudo

这在真实 179 上不成立。

改进：

- 14/15/16 已补充 `user-space pilot` 路径
- 把 `/srv` 正式姿态和 `/home/<user>` 试运行姿态分开写

### 5.2 agent 侧访问方式原先写得过于理想

首轮 `pilot` 没有域名和 nginx，agent 不可能直接访问远端 HTTP。

改进：

- 在快速接入指南里补了 `ssh -L` 访问方式
- 明确 `pilot` 阶段是“访问方式受限”，不是“接入合同变化”

### 5.3 shard 准备责任必须写得更明确

`publish_server_batch.py` 不会帮你自动把全量结果拆成 10 个 reviewer-facing shard。

改进：

- 详细操作手册已明确要求 operator 先准备独立的 shard `result root`

### 5.4 有些步骤不能并行

首轮试运行里出现过一个“看起来像 bug”的假象：

- `review` 返回成功
- 紧接着并行触发 `reviewer bundle export`
- 导出结果出现 `sample_count = 0`

根因不是代码错误，而是执行顺序错了。

改进：

- `publish -> validate`
- `review -> export`

这两类步骤都明确要求串行，不要并行触发

## 6. 下一步建议

1. 继续沿着 `user-space pilot` 路线补一个统一启动/停服/状态检查 wrapper，减少 operator 手敲 `nohup` 的失误面。
2. 在 operator 手册里把“串行步骤”提升成显式 checklist，而不是只放在说明段落里。
3. 再做一轮双 reviewer 或双 agent 的共享试运行，验证多人同时写 reviewer namespace 时的体验与边界。
4. 如果后续要让组内成员直接访问，再由有 sudo 的 operator 把 `/srv + nginx + systemd` 落起来，并保持当前 `datasets/shared/runtime` 目录契约不变。
