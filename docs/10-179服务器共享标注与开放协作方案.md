# 179 服务器共享标注与开放协作方案

## 1. 文档目的

这份文档回答 4 个问题：

1. `trajectory_annotation_studio` 如何部署到 `179` 服务器，承载 `10天 * 10000 UID` 的共享标注。
2. GitHub 代码仓如何开放给组内成员协作，同时继续保持 `零数据 Git`。
3. 大家如何在同一套服务上标注、导出自己的结果，以及连同原始数据一起带走。
4. 大家如何上传自己的数据或方法结果，接入到同一套 studio，而不直接破坏共享批次。

这是一版用于讨论和冻结边界的方案，不是假装“已经全部实现完”的运行说明。

## 2. 当前实现已经具备的基础

当前仓库里的 `trajectory_annotation_studio` 已经具备下面这些真实能力：

- 服务可以按 `--batches-root` 自动发现多个批次并切换。
- 批次可以直接用 `<batch_root>/result`，也可以用 `batch_meta.json.source_result_root` 挂外部结果目录。
- 审核与时间轴标注已经按 `reviewer namespace` 落盘，不再是全局单份。
- 已经支持：
  - `GET/POST /api/reviews`
  - `GET/POST /api/timeline-annotations`
  - `POST /api/export/accepted`
  - `POST /api/export/review-aggregate`
- 已经支持按 `reviewer` 导出 `accepted_assets`，并保留 `raw.csv + line.csv|fmm.csv + review.json`。

这意味着，把它放到 `179` 上的核心工作不是“重做标注平台”，而是：

- 把运行目录、数据目录、权限边界理顺
- 把 `10天 * 10000 UID` 这批数据组织成长期稳定批次
- 把上传/导出/发布流程正规化
- 补上几个当前还缺的服务器级能力

## 3. 总体建议

### 3.1 推荐采用的总体形态

推荐采用：

- `GitHub` 代码仓对协作成员开放
- `179` 服务器承载真实批次、review 写回和导出结果
- 代码仓与服务器数据区彻底分离
- 共享批次采用“发布区 (`published`)”模式，而不是所有人直接写共享根目录
- 用户上传自己的数据时，先进入“接入区 (`incoming`)”，经过校验和发布后再进入共享批次列表

### 3.2 不推荐的做法

不推荐：

- 大家直接 SSH 到线上运行目录里改代码
- 所有人对共享批次根目录有完全写权限
- 把 `10天 * 10000 UID` 直接复制进 GitHub 仓库
- 让用户随意往 `published batch` 里手工加文件

原因很简单：

- 代码会和运行态混在一起
- 批次结构会很快失控
- 导出和回溯会变得不可审计
- 大规模共享标注下，误删和覆盖几乎一定会发生

## 4. 推荐的双层架构

```text
GitHub
  code / docs / adapters / templates / tests

179 Server
  app/                 # 运行中的 studio 代码
  datasets/            # 大体量只读结果数据
  shared/published/    # studio 暴露给大家的批次根
  shared/incoming/     # 大家上传自己数据的接入区
  shared/exports/      # 导出的压缩包或分享副本
  runtime/             # logs / tmp / pid / nginx upload scratch
  workspaces/          # 个人 worktree / 调试副本
```

这里的关键不是目录名字，而是 3 条边界：

1. `app/` 只放代码，不放真实批次。
2. `datasets/` 放大体量只读数据，不让大家直接改。
3. `shared/published/` 只放“已经发布给 studio 的批次入口”和写回目录。

推荐目录如下：

```text
/srv/trajectory_annotation_studio/

  shared/

      signal100_10d_uid10000_round1_b01of10/

  runtime/
    logs/

  workspaces/
    ocean/
    alice/
    bob/
```

这里还要额外强调一条：

- `incoming/` 不应放在 `--batches-root` 扫描根下面

因为当前 `review_server.py` 会扫描 `batches-root` 下一层子目录，只要能解析出真实结果目录，就可能把它当成可见批次挂出来。

### 5.1 为什么 `published batch` 里不一定要放 `result/`

这是当前实现的一个重要优点。

`trajectory_annotation_studio/web/review_server.py` 已支持：

- 优先读 `<batch_root>/result`
- 若不存在，则读 `batch_meta.json.source_result_root`

所以发布到共享标注页的批次可以是：

```json
{
  "name": "signal100_10d_uid10000_round1",
  "label": "signal100_10d_uid10000_round1",
  "version": "v1",
  "created_at": "2026-04-20T08:00:00Z",
  "keywords": ["signal100", "10d", "uid10000", "round1", "shared"],
  "source_result_root": "/srv/trajectory_annotation_studio/datasets/cellular_quality/signal100_10d_uid10000_round1/result",
  "result_mode": "mounted",
  "uid_count": 10000
}
```

这样能把：

- 大结果数据目录
- 共享 review 写回目录
- 导出目录

分开管理，后续权限更清楚。

## 6. GitHub 开放策略

## 6.1 推荐策略

推荐先采用：

- `GitHub 私有仓 + 开放协作成员`

也就是：

- 代码仓对组内参与者开放
- 真实数据和服务器路径不进仓
- 运维私密配置不进仓

这是当前最稳妥的第一阶段。

## 6.2 为什么不建议一开始就完全 public

如果一开始就做成完全公开仓，通常会立刻遇到 3 个问题：

1. 文档里会混进服务器路径、内网地址、部署细节，需要再清洗。
2. 很多“仅组内使用”的上传/运维说明并不适合公开暴露。
3. 你现在更需要“让组内能协作”，而不是“让外部陌生人立刻参与”。

因此更推荐两阶段：

### Phase A

- 私有仓
- 开给组内成员、agent 开发者、方法贡献者

### Phase B

若后续希望公开代码：

- 拆出公开可见的 core/docs
- 把 179 服务器私密部署文档抽离
- 清洗路径、内部主机名、上传入口说明

## 7. 角色与权限模型

建议固定 5 类角色：

### 7.1 Maintainer（代码维护者）

负责：

- GitHub PR 审核
- `main` 分支维护
- 发布代码到 `179 app/current`
- 冻结批次合同和导出合同

### 7.2 Operator（平台运行人）

负责：

- 在 `179` 上发布共享批次
- 运行接入校验
- 维护导出任务
- 管理 `published / incoming / exports`

### 7.3 Reviewer（标注者）

负责：

- 通过 Web 标注和审核
- 只在自己的 reviewer 命名空间下写入
- 导出自己的审阅结果

### 7.4 Data Uploader（数据接入者）

负责：

- 把自己要接入的数据上传到 `incoming`
- 提供 adapter、manifest、字段说明
- 配合完成发布校验

### 7.5 Participant（普通使用者）

负责：

- 使用已发布批次
- 提交代码或方法
- 不直接操作服务器共享数据根

## 8. Linux 权限建议

## 8.1 推荐用户/组

推荐至少有下面这些系统身份：

- 服务用户：`ta_svc`
  说明：运行 `review_server.py`、写 review/导出目录
- 管理组：`ta_admin`
  说明：维护 app、发布批次、做清理和恢复
- 审阅组：`ta_reviewers`
  说明：只需要 Web 访问，不一定需要 SSH
- 上传组：`ta_uploaders`
  说明：允许上传自己的接入数据到 `incoming`

## 8.2 目录权限原则

### `app/`

- `ta_svc:ta_admin`
- 仅 `ta_admin` 可写
- 普通参与者不应直接改这里

### `datasets/`

- 只读为主
- `ta_admin` 可写
- `ta_svc` 只读
- 其他用户默认不直接写

### `shared/published/`

- `batch_meta.json`、`review/`、`accepted_assets/`、`review_exports/` 由 `ta_svc` 维护
- 用户不直接 SSH 改共享批次
- 大家通过 Web 交互写入 reviewer 命名空间

### `shared/incoming/`

- 每个用户一个子目录
- 上传者对自己的目录可写
- `ta_admin` 和 `ta_svc` 可读
- 推荐加 `setgid`，保证组继承

### `shared/exports/`

- 由 `ta_svc` 生成
- 下载给用户时可只读

## 8.3 权限结论

对于“是否需要改部分文件夹权限”这个问题，答案是：

- 需要
- 但不应该通过“所有人对 `published` 全部可写”来解决

推荐做法是：

- `incoming` 给上传者写权限
- `published` 交给服务进程写
- `datasets` 尽量只读

这是共享场景下最稳的权限切法。

## 9. 179 上的服务形态

推荐：

- `systemd` 托管 `review_server.py`
- `nginx` 或 `caddy` 做反向代理
- 反向代理层负责：
  - HTTPS
  - 基本访问控制
  - 上传体积限制
  - 日志

当前 `review_server.py` 本身没有：

- 账号登录
- 权限鉴别
- 上传 API

所以如果 179 是多人共享环境，外层代理必须负责第一层保护。

另外，当前实现更适合：

- 单机
- 单服务实例
- 单进程写共享文件系统

不建议一上来就做多 worker、多实例共享写同一批次目录。当前 `review_lib.py` 的锁是进程内锁，不是跨进程分布式锁。

## 10. 10天 * 10000 UID 共享批次如何组织

推荐把这批数据分成两个层次看：

### 10.1 一个冻结 cohort

也就是：

- `signal100_10d_uid10000_round1`

它代表的是：

- 同一批 `10000 UID`
- 同一组 `10天`
- 同一版算法产物
- 同一版 `manifest/states_index`

这个 cohort 是“数据事实源”。

### 10.2 十个 reviewer-facing shard

推荐不要让 reviewer 首轮直接面对一个单批 `10000 UID`。

更推荐：

- `signal100_10d_uid10000_round1_b01of10`
- `signal100_10d_uid10000_round1_b02of10`
- ...
- `signal100_10d_uid10000_round1_b10of10`

每个 shard 大约 `1000 UID`。

原因：

- 当前批次发现逻辑是平铺的一层目录，天然更适合 shard 级管理
- reviewer 选择、分工、导出、关闭批次都会更清楚
- 后续要做“某批已审完、某批继续开”的治理也更容易

所以推荐形态是：

- `datasets/` 里保留一个完整 `uid10000` cohort
- `shared/published/` 里发布 `10` 个 shard 批次入口

例如：

```text
/srv/trajectory_annotation_studio/datasets/cellular_quality/
  signal100_10d_uid10000_round1/

/srv/trajectory_annotation_studio/shared/published/
  signal100_10d_uid10000_round1_b01of10/
  signal100_10d_uid10000_round1_b02of10/
  ...
  signal100_10d_uid10000_round1_b10of10/
```

### 10.3 shard 的 `batch_meta` 推荐补充字段

每个 shard 建议补这些字段：

- `cohort_id`
- `shard_index`
- `shard_count`
- `days`
- `status`
- `uid_manifest_checksum`

例如：

```json
{
  "name": "signal100_10d_uid10000_round1_b01of10",
  "label": "signal100_10d_uid10000_round1_b01of10",
  "version": "v1",
  "cohort_id": "signal100_10d_uid10000_round1",
  "shard_index": 1,
  "shard_count": 10,
  "days": ["2023-03-01", "2023-03-02", "2023-03-03", "2023-03-04", "2023-03-05", "2023-03-06", "2023-03-07", "2023-03-08", "2023-03-09", "2023-03-10"],
  "status": "published",
  "uid_count": 1000
}
```

## 11. 共享标注工作流

## 11.1 Reviewer 的日常工作流

1. 打开 `179` 上的 studio。
2. 选择批次 `signal100_10d_uid10000_round1`。
3. 输入自己的 reviewer 身份。
4. 在自己的 reviewer 命名空间下做：
   - 整轨审核
   - 时间轴标注
5. 按需导出自己的结果。

### 当前代码已经支持的部分

- reviewer 命名空间
- 多 reviewer 汇总
- per-reviewer accepted 导出

### 当前还缺的部分

- “一键导出我全部已标注 UID + 原始数据”的完整 reviewer bundle

这个能力建议作为 179 方案的首批增强项。

## 11.2 推荐新增的导出模式

为了满足“大家可以复制/导出自己标注的数据 + 对应原数据”，建议新增两类导出：

### A. `accepted export`

当前已具备。

内容：

- 只导出当前 reviewer 最新为 `accept` 的 UID
- 附带：
  - `raw.csv`
  - `line.csv` 或 `fmm.csv`
  - `review.json`

适合：

- 构建高质量参考集

### B. `reviewer bundle export`

这是建议新增的能力。

内容：

- 导出当前 reviewer 所有已标注 UID
- 每个 UID 可附带：
  - `raw.csv`
  - `snap.csv`
  - `od.csv`
  - `fmm.csv`
  - `line.csv`
  - `review.json`
  - `timeline_annotations.json`

建议目录：

```text
exports/reviewers/<reviewer_id>/bundles/<bundle_name>/
  bundle_manifest.json
  reviews/
    latest_reviews.json
    ledger.jsonl
  timeline_annotations/
    <uid>.json
  samples/
    <uid>/
      raw.csv
      snap.csv
      od.csv
      fmm.csv
      line.csv
      review.json
      timeline_annotations.json
```

适合：

- 个人带走自己的全部工作结果
- 后续离线分析
- 训练和复审

## 12. 用户上传自己数据的推荐流程

不建议用户直接把自己的东西扔进 `published`。

推荐固定成 4 步：

### Step 1. 上传到 `incoming`

用户通过：

- `scp`
- `sftp`
- Web 上传页（后续能力）

把数据传到：

```text
/srv/trajectory_annotation_studio/shared/incoming/<user>/<upload_name>/
```

### Step 2. 标记类型

上传类型分两类：

#### Type A：已是标准批次

用户已经准备好：

- `batch_meta.json`
- `result/manifest.json`
- `result/<uid>/...`

这种只需要做结构校验和发布。

#### Type B：原始自定义数据

用户上传的是自己的原始数据、方法输出或特殊格式，需要：

- 在 GitHub 仓库里提供 `adapter`
- 由 adapter 转成标准批次结构

### Step 3. 运行发布前校验

Operator 运行接入校验，至少检查：

- `manifest.json` 是否存在
- `uids` 是否可解析
- 关键 CSV 是否存在
- `batch_meta` 是否完整
- 是否包含非法大文件、路径穿越、缓存垃圾

### Step 4. 发布到 `published`

发布时不一定复制大数据。

更推荐：

- 在 `published/<batch>/batch_meta.json` 里写 `source_result_root`
- `review/accepted_assets/review_exports` 留在 `published/<batch>/`

这样：

- 结果数据还在自己的数据目录
- studio 能把它作为一个正式批次挂出来

## 13. 推荐的批次生命周期

建议固定 5 个状态：

1. `incoming`
2. `validated`
3. `published`
4. `archived`
5. `retired`

### `incoming`

仅表示已经上传，尚未挂到 studio。

### `validated`

结构已经通过，但还未正式向大家开放。

### `published`

已经出现在 studio 批次列表，可供 reviewer 使用。

### `archived`

不再默认展示，但目录和写回结果保留。

### `retired`

彻底下线，只保留归档包或长期备份。

## 14. 代码发布流程

推荐流程：

1. 大家在 GitHub 提交 PR。
2. `main` 合并后，Maintainer 在 179 上生成一个新 release 目录。
3. 通过 `app/current` symlink 切换到新 release。
4. 重启 `systemd` 服务。
5. 共享批次目录不动，继续沿用。

这样能避免：

- 代码升级把 `review/` 写回弄丢
- 大家在运行目录里直接改代码

## 15. 当前实现缺口

要达到你设想的完整形态，当前 repo 还缺 5 件事：

1. 服务器级鉴权边界
   说明：当前 `review_server.py` 无鉴权，需要交给 `nginx/caddy + 内网/VPN/basic auth`。

2. 用户上传入口
   说明：现在已经有第一版 upload API 和 `admin_upload.html`，但还没有细粒度审批 UI、断点续传和完善权限。

3. `reviewer bundle export`
   说明：现在已经有第一版 bundle export，能导出 reviewer 的审核、时间轴标记与对应样本工件；后续还可补更细的筛选与下载治理。

4. 发布前批次校验器
   说明：现在已经有 `validate_published_batch.py`、`publish_server_batch.py` 与第一版 `process_incoming_upload.py`；后续还可补更完整的 intake admin 工作台。

5. 统一标签目录服务端化
   说明：当前自定义标签仍带 `localStorage` 色彩，不适合作为强治理能力。

## 16. 推荐实施顺序

推荐按下面顺序推进：

### Phase 1：179 基础部署

- 准备 `app/current`
- 准备 `datasets/`
- 准备 `shared/published`、`shared/incoming`、`shared/exports`
- 用 `systemd + nginx` 起服务

### Phase 2：发布第一轮 `10天 * 10000 UID`

- 在 `datasets` 准备完整 `signal100_10d_uid10000_round1/result`
- 先切成 `10` 个 `1000 UID` shard
- 在 `published` 建 `10` 个对应批次入口
- 每个 shard 用 `source_result_root` 挂到 studio

### Phase 3：补导出闭环

- 增加 `reviewer bundle export`
- 支持把“我的标注 + 对应原数据”打成可下载包

### Phase 4：补上传接入闭环

- 增加 `incoming -> validate -> publish` 脚本
- 再决定是否做 Web 上传页

### Phase 5：再讨论 public 化

- 如果之后要把 GitHub 完全公开，再清洗运维文档和私有路径

## 17. 本轮建议冻结的决定

为了后续不反复改方向，建议先冻结下面 6 条：

1. GitHub 第一阶段采用“私有仓 + 开给协作成员”，不是立刻完全 public。
2. `10天 * 10000 UID` 冻结为一个 cohort，但对 reviewer 侧发布为 `10` 个 `1000 UID` shard，而不是单批 `10000 UID` 全开。
3. 共享批次采用 `published batch + source_result_root`，而不是把大数据直接塞进批次目录。
4. 上传采用 `incoming -> validate -> publish`，而不是所有人直接写共享批次。
5. 权限只给上传区，不给所有人共享批次写权限。
6. 首批新增能力优先做 `reviewer bundle export`，因为这直接关系到大家能否带走自己的标注成果。
